## 2. 实盘工程风险清单

| 风险 | 后果 |
| --- | --- |
| 配置错误 | 错误费率、错误风控、错误 symbol 状态 |
| 行情缺包 | 本地订单簿错误，策略误判 |
| 行情延迟 | 基于过期价格下单 |
| 重连失败 | 长时间失去交易或行情能力 |
| 重复下单 | 资金重复占用或非预期仓位 |
| 撤单失败未处理 | 订单残留在市场中 |
| 部分成交处理错误 | 持仓和订单状态不一致 |
| 日志不足 | 事故后无法定位 |
| 指标缺失 | 线上异常发现太晚 |
| 报警过多 | 真正问题被噪声掩盖 |
| 对账缺失 | 账不平长时间未发现 |
| 人工操作无审计 | 难以追责和恢复 |

## 3. 配置管理

交易系统配置包括：

- symbol 状态
- tickSize / lotSize / minNotional
- 费率
- 风控限额
- API 权限
- 限频参数
- 市价保护比例
- 交易开关
- 灰度策略

配置必须具备：

```text
版本
审批
生效时间
回滚能力
审计日志
```

不要让关键配置通过手工改数据库生效。

## 4. 行情一致性

行情系统最怕本地订单簿不完整。

关键要求：

- snapshot 有序号
- incremental 有序号
- 缺口能发现
- 能重新同步
- 延迟可监控
- 消息处理可幂等

如果本地订单簿错了，策略和用户看到的市场都是错的。

## 5. 日志

实盘日志要能回答：

```text
谁在什么时候做了什么？
系统收到什么？
系统发出了什么？
为什么拒绝？
为什么成交？
为什么撤单失败？
```

核心日志：

- 请求日志
- 订单状态日志
- 成交事件日志
- 清算日志
- 账本日志
- 风控拒绝日志
- 重连日志
- 配置变更日志

热路径日志要谨慎，不能为了可观测性拖慢撮合。

## 6. 重连和补偿

实盘一定会断线。

断线恢复要处理：

- 会话重新登录
- 序号对齐
- 未收到回报补查
- open orders 查询
- fills 查询
- balance 查询
- 本地状态修正

不要假设：

```text
网络永远稳定
交易所回报永远不丢
撤单一定成功
```

## 7. 线上最想先看到的 10 个监控指标

| 指标 | 说明 |
| --- | --- |
| 下单 QPS | 当前请求压力 |
| 下单成功率 / 拒单率 | 判断业务和系统异常 |
| 撮合延迟 P50/P99/P99.9 | 核心处理稳定性 |
| 行情延迟 | 撮合到行情发布耗时 |
| 行情缺口次数 | 本地簿或发布链路异常 |
| 队列积压长度 | 下游是否消费变慢 |
| 订单回报延迟 | 用户感知延迟 |
| 清算延迟 | 成交到账务耗时 |
| 对账差异数 | 状态一致性风险 |
| 错误码分布 | 快速定位异常类型 |

额外建议：

- GC pause
- CPU 使用率
- 内存分配率
- 网络重连次数
- Aeron back pressure 次数
- snapshot 耗时
- replay 耗时

## 8. 报警

报警不是越多越好。

高价值报警应该满足：

```text
需要人工介入
影响交易正确性或可用性
有明确处理手册
能定位到模块和影响范围
```

低价值报警会让团队疲劳，最终忽视真正事故。

## 9. 业务错误还是工程错误

交易系统线上问题通常是混合的。

业务错误：

- 订单类型理解错
- 手续费规则错
- 市价单保护错
- 结算价规则错
- 自成交防护策略错

工程错误：

- 消息乱序
- 幂等缺失
- 缺包未发现
- 重连恢复失败
- 并发更新余额
- 事件重复消费

很多事故表面是业务错误，本质是工程边界没设计好。

## 11. 复盘问题

交易系统最常见的问题更偏业务错误还是工程错误？

可以这样回答：

交易系统问题通常是业务规则和工程实现交织在一起。订单类型、清算、风控和结算属于业务规则；序号、幂等、重放、延迟、重连和并发属于工程实现。生产事故往往不是单纯某一类错误，而是业务规则没有被工程边界正确承载，例如重复成交、冻结释放错误、行情缺包未发现、订单状态乱序等。
