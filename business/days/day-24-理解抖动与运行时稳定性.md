# Day 24：理解抖动与运行时稳定性

## 1. 今天的学习目标

## 2. 核心术语

### 2.1 平均延迟

平均延迟是所有请求延迟的平均值。

例如：

```text
1000 个请求平均延迟 = 1ms
```

它能反映整体水平，但掩盖极端慢请求。

### 2.2 P99 延迟

`P99` 表示 99% 的请求延迟都小于等于这个值。

例如：

```text
P99 = 8ms
```

含义是：

```text
1000 个请求里，大约 990 个请求延迟 <= 8ms
```

### 2.3 尾延迟

尾延迟是高分位延迟，例如：

```text
P99
P99.9
P99.99
max
```

交易系统关注尾延迟，因为极少数慢请求也可能造成用户重大损失。

### 2.4 抖动

抖动是延迟不稳定。

例如：

```text
大部分请求 1ms
偶尔跳到 50ms
```

这比稳定的 3ms 更难处理，因为策略和用户无法预期。

## 3. 吞吐量、平均延迟、P99 的区别

| 指标 | 含义 | 例子 |
| --- | --- | --- |
| 吞吐量 | 单位时间能处理多少请求 | 每秒 100000 单 |
| 平均延迟 | 请求平均处理时间 | 平均 1ms |
| P99 延迟 | 99% 请求能在多久内完成 | P99 8ms |
| 最大延迟 | 最慢请求花了多久 | max 200ms |

高吞吐不等于低延迟。

一个系统可能：

```text
吞吐很高，但排队严重，P99 很差
```

低延迟系统追求的是：

```text
高吞吐 + 稳定低尾延迟
```

## 4. 交易系统为什么更怕长尾延迟

交易系统里的慢请求会造成：

- 用户订单比预期晚进入撮合
- 撤单慢导致被动成交
- 行情慢导致本地订单簿错误
- 风控慢导致订单排队
- 清算慢导致余额更新延迟
- 下游消费慢导致事件积压

平均延迟好看不能说明系统稳定。

例子：

```text
系统 A:
  平均延迟 1ms
  P99 100ms

系统 B:
  平均延迟 3ms
  P99 5ms
```

对交易系统来说，系统 B 可能更可靠。

## 5. 抖动来源

常见抖动来源：

| 来源 | 说明 |
| --- | --- |
| GC | Stop-the-world 或分配压力导致暂停 |
| 锁竞争 | 多线程争用共享资源 |
| CPU 调度 | 关键线程被操作系统切走 |
| 上下文切换 | 线程过多或阻塞唤醒频繁 |
| 缓存未命中 | CPU cache miss |
| 日志 IO | 同步写日志或日志刷盘 |
| 网络抖动 | TCP 重传、网卡中断、拥塞 |
| 磁盘 IO | journal 或数据库写入不稳定 |
| 动态扩容 | HashMap 扩容、数组扩容 |
| 序列化 | JSON、大对象转换、反射 |

## 6. 低延迟调优点清单

| 方向 | 调优点 |
| --- | --- |
| 对象分配 | 热路径减少 new、复用 buffer |
| 数字类型 | 价格数量用 long 定点数，避免 BigDecimal 高频运算 |
| 日志 | 热路径避免 INFO 日志和 JSON 序列化 |
| 数据结构 | 避免频繁扩容，选择适合订单簿的结构 |
| 线程模型 | 单 symbol 单线程或分片有序处理 |
| 锁 | 避免撮合热路径锁竞争 |
| GC | 控制分配率，选择合适 GC 策略 |
| CPU | 关键线程绑核，隔离核心 |
| IO | journal 顺序写，避免随机 IO |
| 网络 | 使用低延迟传输和背压策略 |
| 指标 | 记录 P50/P90/P99/P99.9/max |
| 回放 | 回放路径和线上处理路径尽量一致 |

### 6.1 背压策略

背压是系统在下游处理不过来时，主动限制上游继续灌入流量的机制。

如果没有背压，系统通常不会立刻失败，而是先堆积：

```text
请求队列越来越长
内存占用越来越高
GC 压力越来越大
P99 / P99.9 延迟急剧变差
最终出现超时、OOM、断连或大面积重试
```

交易系统里，背压的目标不是“所有消息都排队等着处理”，而是保护核心链路保持可预测：

```text
能处理的请求快速处理
不能处理的请求尽早拒绝、限流、降级或转入可靠重试
不要让慢下游拖垮撮合线程
```

常见背压策略：

| 策略 | 含义 | 适用场景 |
| --- | --- | --- |
| 限流 | 在入口控制请求速率 | Gateway、API、会话层 |
| 有界队列 | 队列达到上限后拒绝或阻塞生产者 | OMS、Outbox、行情发布 |
| 快速失败 | 系统繁忙时直接返回明确错误 | 下单、改单、撤单入口 |
| 重试退避 | 上游失败后延迟重试，避免雪崩 | OMS 到 Matching、清算消费 |
| 丢弃非关键消息 | 对可重建或低优先级消息丢弃 | 行情广播、监控指标 |
| 降级 | 暂停部分非核心功能 | 报表、审计查询、非关键推送 |
| 隔离 | 慢消费者不能阻塞快消费者 | 行情、清算、用户回报 |

不同链路的处理方式不同。

订单入口：

```text
Gateway / Session 队列满
  -> 限流或拒单
  -> 返回系统繁忙错误
  -> 不要无限排队
```

OMS 到 Matching：

```text
Aeron offer 出现背压
  -> Dispatcher 短暂 spin / park
  -> 记录 in-flight 和发送延迟
  -> 超过阈值后停止继续取新 outbox
  -> 依靠 outbox 做恢复和重试
```

撮合结果输出：

```text
MatchResult 生成后
  -> 可靠结果日志必须保留
  -> 慢下游不能阻塞撮合主线程太久
  -> Publisher 可以异步读取结果日志再分发
```

行情广播：

```text
普通行情推送可以丢弃旧增量或断开慢客户端
但行情系统必须保留可恢复的快照 + 增量序号
客户端发现缺口后重新拉快照
```

清算和账本：

```text
清算事件不能随意丢弃
如果清算慢，应积压在可靠队列或事件日志中
同时监控积压长度和最大滞后时间
必要时限制新订单入口
```

对当前项目来说，尤其要关注：

```text
ResultMdcBroadcaster 的 offer 背压不能长时间阻塞撮合状态机。
如果广播失败或下游慢，应该优先保证 MatchResult 已进入可靠结果存储，再由独立 Publisher 重放和分发。
```

## 7. Java 项目特别要注意什么

当前项目已经有一些低延迟优化方向：

- SBE 二进制编码
- Aeron Cluster
- V1 中价格数量用 `long`
- Agrona 集合
- 减少热路径日志

还应持续关注：

- `BigDecimal` 是否仍在热路径
- `LinkedList` 是否造成指针追逐
- `log.info` 是否出现在高频撮合路径
- `JSON` 序列化是否在热路径
- `Map` 是否频繁扩容
- result list 是否频繁分配
- snapshot 是否影响主链路
- result broadcast 背压是否会阻塞撮合

## 8. 线程绑定与 CPU 隔离

线程绑定是把关键线程固定在特定 CPU core 上。

目的：

- 减少操作系统调度抖动
- 提高 CPU cache 命中
- 避免关键线程和普通线程抢 core

CPU 隔离是把某些 core 留给关键线程。

典型做法：

```text
core 0-1: OS / 普通服务
core 2: gateway
core 3: sequencer
core 4: matching BTC shard
core 5: matching ETH shard
core 6: market data publisher
```

这类优化通常在系统设计稳定后再做，不应该过早替代正确架构。

## 9. 小练习

区分吞吐量、平均延迟、P99 延迟分别意味着什么。

参考回答：

```text
吞吐量表示系统单位时间能处理多少请求。
平均延迟表示所有请求耗时的平均水平。
P99 表示 99% 请求能在该时间内完成，反映尾部稳定性。
```

## 10. 复盘问题

为什么交易系统更怕长尾延迟而不是平均延迟略高？

可以这样回答：

交易系统中的少数慢请求也可能造成严重后果，例如撤单延迟导致非预期成交、行情延迟导致错误下单、清算延迟导致资金状态滞后。平均延迟会掩盖这些极端慢请求，而 P99、P99.9 和最大延迟更能反映系统是否稳定。交易系统更需要可预测的低尾延迟，而不是只追求漂亮的平均值。
