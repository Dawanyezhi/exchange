# 撮合一致性与恢复生产改进方案

## 1. 背景

当前项目的 `matching` 已经具备撮合系统走向生产化的一些关键基础：

```text
Aeron Cluster 输入日志
单线程撮合状态机
request serialNum 连续性校验
共识时间戳
撮合状态快照
resultSerialNum 撮合结果序号
MatchResult 结果广播
```

这些能力可以保证撮合引擎自身在单撮合集群、单输入流模型下具备确定性处理和基础恢复能力。

但生产交易系统的一致性与恢复不只要求“撮合状态可恢复”，还要求：

```text
输入命令可追溯
撮合结果可可靠重放
下游消费可幂等
快照与日志边界一致
故障后能从明确位置继续
OMS、清算、行情、账本都能解释同一套事实
```

## 2. 一致性的核心目标

生产交易系统的一致性不是所有模块必须同步完成，而是所有事实最终能互相解释。

需要满足：

```text
订单状态能被订单事件解释
成交账能被撮合事件解释
账户余额能被账本流水解释
行情深度能被订单簿事件解释
清算结果能被成交和订单终态事件解释
```

撮合系统在其中承担两个核心职责：

```text
1. 对进入订单簿的命令给出确定执行顺序。
2. 输出可恢复、可重放、可幂等消费的撮合结果事件。
```

## 3. 当前项目已有能力

### 3.1 输入侧顺序

当前项目通过 `serialNum` 做全局连续性校验：

```text
newSerialNum == lastSerialNum + 1
```

在单撮合集群模型下，这是合理的。

优点：

- 实现简单。
- 所有请求有全局顺序。
- 快照只需要保存一个 `lastSerialNum`。
- 恢复后小于等于 `lastSerialNum` 的请求可以视为重复。

局限：

- 所有 symbol 共享一个顺序。
- 任意一个序号缺口会阻塞所有交易对。
- 未来拆多撮合分片时需要升级为 `partitionId + partitionSeq`。

多撮合集群按 symbol 或 partition 拆分后，好处是不同交易对不会互相阻塞：

```text
BTC-USDT 的序号缺口只影响 BTC-USDT
ETH-USDT 可以继续按自己的 partitionSeq 撮合
```

这样可以提升横向扩展能力、降低热门交易对对其他交易对的尾延迟影响，并且让故障恢复、快照、重放和迁移都可以按分区处理。

代价是路由、序号、快照、结果流和下游消费都会更复杂。因此当前阶段使用全局 `serialNum` 是合理的，等吞吐、交易对数量或故障隔离需求上来后，再升级到：

```text
globalCommandId + partitionId + partitionSeq
```

### 3.2 状态机恢复

当前项目使用 Aeron Cluster 管理输入日志和状态机回放，快照中保存：

```text
maxProcessedRequestSerialNum
maxResultSerialNum
symbol config
order book
open orders
checksum
```

这是撮合系统恢复的基础。

### 3.3 结果事件

当前项目通过 `MatchResult` 输出撮合结果，并通过 `resultSerialNum` 做结果事件序号。

这可以作为下游消费幂等和顺序检查的基础。

## 4. 当前不足

### 4.1 结果日志还不是严格 durable commit

当前结果持久化使用 Aeron Publication 写入 Archive stream。

但 `offer` 成功通常只代表写入了 publication buffer，不严格等于：

```text
Archive 已经记录到指定 position
磁盘已经可靠落盘
重启后一定能从该位置 replay
```

生产要求：

```text
如果 MatchResult 是清算、OMS、行情的事实源，
则必须能确认 result event 已进入可靠结果日志。
```

改进方向：

```text
记录 publication position
append 路径不逐条等待 Archive recording position 追上
同步快照前等待 Archive recording position 追上快照边界对应的 lastOfferedPosition
resultSerialNum -> recordingPosition 索引由独立 replay-service 异步构建
启动时校验 result log 连续性
```

### 4.2 结果广播与撮合状态机耦合

此前 `flushAndPersist()` 在撮合状态机线程中执行：

```text
生成 MatchResult
持久化旧 ResultRepository
matching 内 MDC 广播
```

当前已完成第一步改造：

```text
撮合线程只写可靠结果日志
matching 不再内置 MDC 广播
matching 不再提供 replay 查询
```

生产最终形态：

```text
独立 Result Publisher 从结果日志读取
再推送 OMS / Clearing / MarketData
```

原因：

- 慢下游不能阻塞撮合。
- 广播背压不能影响订单簿更新。
- 下游断线后可以从结果日志 replay。
- leader 切换后 Publisher 可以从 checkpoint 继续。

当前代码边界：

```text
CommandDispatcher
  -> MatchResultEventsHelper.endRequest()
  -> ResultLogWriter.append(batch)
  -> ArchiveResultLogWriter
  -> Aeron Archive result log
```

读取、实时推送和 replay 查询留给后续独立 `result-publisher` 服务实现。

### 4.3 下游 replay 机制不足

实时广播可以丢，但可靠结果日志不能丢。

生产下游需要：

```text
记录 lastConsumedResultSerialNum
发现 resultSerialNum 缺口
主动请求 replay
从 result log 指定序号恢复
幂等处理重复事件
```

当前项目还需要补：

```text
Result Replay Service
consumer checkpoint
缺口检测
result archive 查询接口
下游恢复流程
```

### 4.4 幂等边界需要更清晰

当前可以用全局 `serialNum` 保证输入流顺序，用 `orderId` 防重复下单，用 `resultSerialNum` 防重复消费。

这在当前阶段可行。

生产中建议把不同身份拆开：

| ID | 含义 | 是否当前必须 |
| --- | --- | --- |
| `orderId` | 订单实体身份 | 必须 |
| `requestSerialNum` / `sequencerId` | 进入撮合输入流的顺序身份 | 必须 |
| `resultSerialNum` | 撮合结果事件身份 | 必须 |
| `commandId` | 一次下单、改单、撤单操作身份 | 建议预留 |
| `matchId` / `tradeId` | 一笔真实成交事实身份 | 生产建议 |
| `orderVersion` | 订单状态版本 | 改单/撤单建议 |

这些 ID 不是互相替代，而是表达不同语义。

## 5. 统一幂等模型

清算和账本不应该只围绕 `matchId` 设计，而应该围绕“来源事件”设计。

统一幂等键：

```text
sourceType + sourceId + accountId + asset + entryType
```

含义：

| 字段 | 含义 |
| --- | --- |
| `sourceType` | 来源事件类型，例如 MATCH、CANCEL、REJECT、EXPIRE |
| `sourceId` | 来源事件 ID，当前可用 resultSerialNum |
| `accountId` | 账户 |
| `asset` | 资产 |
| `entryType` | 流水类型，例如 TRADE_DEBIT、TRADE_CREDIT、FEE、RELEASE_FROZEN |

数据库唯一约束：

```sql
unique(source_type, source_id, account_id, asset, entry_type)
```

### 5.1 有成交

当前阶段可以使用：

```text
sourceType = MATCH
sourceId = resultSerialNum
```

生产增强后可以增加：

```text
matchId = 真实成交事实 ID
sourceEventId = resultSerialNum
```

清算可以按 `matchId` 聚合成交事实，同时账本仍保留 `resultSerialNum` 用于追溯来源事件。

### 5.2 无成交

无成交时没有 `matchId`。

例如：

```text
挂单成功
改单成功
撤单成功
拒单
订单过期
市价单残余释放
```

这类事件如果需要清算或账本动作，使用对应事件的 `resultSerialNum` 作为 `sourceId`。

示例：

```text
撤单释放冻结:
  sourceType = CANCEL
  sourceId = cancelResultSerialNum
  entryType = RELEASE_FROZEN

拒单释放冻结:
  sourceType = REJECT
  sourceId = rejectResultSerialNum
  entryType = RELEASE_FROZEN

市价单残余释放:
  sourceType = MARKET_REMAINDER_RELEASE
  sourceId = terminalResultSerialNum
  entryType = RELEASE_REMAINING_BUDGET
```

这样有成交和无成交都走同一套账本幂等模型。

## 6. commandId 是否必须

当前阶段，`PLACE` 使用 `orderId` 做幂等是可以的。

但生产中建议预留 `commandId`，因为：

```text
orderId 表示订单实体
commandId 表示一次操作请求
requestSerialNum 表示撮合输入流顺序
```

同一个订单可能有多次操作：

```text
PLACE  orderId=1001
AMEND  orderId=1001
AMEND  orderId=1001
CANCEL orderId=1001
```

如果只看 `orderId`，无法区分：

```text
这是同一次撤单请求重试？
还是一次新的撤单请求？
这是第一次改单？
还是第二次改单？
```

推荐：

```text
PLACE:
  orderId 可作为主要幂等键
  commandId 作为请求追踪和重复投递保护

AMEND:
  commandId + orderId + expectedOrderVersion

CANCEL:
  commandId + orderId
```

当前项目可以先不强制引入 `commandId`，但协议和表结构建议预留。

## 7. orderVersion 的作用

`requestSerialNum = lastRequestNum + 1` 只能保证输入流顺序连续。

它解决的是：

```text
这条命令是不是按撮合输入顺序到达？
```

但它不能完全解决：

```text
这条改单/撤单命令是否仍然适用于当前订单状态？
```

示例：

```text
orderVersion = 3
收到 AMEND expectedOrderVersion = 1
```

即使该请求的 `requestSerialNum` 连续，它也应该被拒绝，因为它基于旧订单状态。

生产规则：

```text
requestSerialNum 连续 -> 允许进入状态机
orderId 存在 -> 找到订单
expectedOrderVersion 匹配 -> 允许改单/撤单
订单未终态 -> 执行
否则 -> reject / already terminal
```

## 8. matchId 的定位

`matchId` 不是订单号，也不是事件序号。

它表示一笔真实成交事实：

```text
makerOrderId
takerOrderId
price
baseQty
quoteQty
tradeTime
```

没有真实成交时，不应该生成 `matchId`。

当前阶段：

```text
一个 MatchOrderResult = 一笔真实成交
resultSerialNum 全局唯一
清算可以用 resultSerialNum 做幂等
```

生产增强：

```text
MatchOrderResult 增加 matchId / tradeId
清算按 matchId 聚合成交事实
账本仍用 sourceType + sourceId + accountId + asset + entryType 做唯一约束
```

## 9. 快照与日志一致性要求

生产快照不只是保存订单簿。

还应保存：

```text
snapshotVersion
schemaVersion
maxProcessedRequestSerialNum
maxResultSerialNum
commandLogPosition
resultLogPosition
configVersion
checksum
createdTime
```

恢复时需要校验：

```text
快照完整
checksum 正确
schema version 可兼容
command log 从快照位置之后连续
result log 从快照位置之后连续
nextResultSerialNum 不回退
```

当前项目已经有 checksum 和核心序号，但还需要补强 result log position、configVersion 和更明确的恢复校验流程。

## 10. OMS 到 Matching 的一致性

生产中不建议 OMS 内存处理完后直接发送撮合命令。

推荐：

```text
OMS / Risk / Account DB Transaction
  -> clientOrderId 幂等
  -> 创建或更新订单状态
  -> 风控通过
  -> 资金冻结
  -> 生成全局 sequencerId
  -> 写 matching_command_outbox
  -> commit

Outbox Dispatcher
  -> 按 sequencerId 严格递增发送 Matching

Matching
  -> Aeron Cluster 记录输入日志
  -> SerialNumValidator 校验连续性
  -> 执行撮合
  -> 输出 MatchResult
```

当前阶段可以继续使用全局 `sequencerId`。

未来多撮合分片时再升级为：

```text
globalCommandId
partitionId
partitionSeq
```

## 11. 控制面命令要求

上下币、交易开关、撮合参数调整也会影响撮合结果。

生产中控制面命令也应具备：

```text
controlCommandId
controlSeq
configVersion
operator
effectiveTime
auditReason
```

否则回放和审计时难以解释：

```text
为什么某个时间点 symbol 不可交易？
为什么某笔订单被拒绝？
当时使用的是哪个撮合配置？
```

## 12. 下游恢复要求

OMS、清算、行情、账本都应该保存自己的消费位点。

示例：

```text
consumerName = clearing
lastConsumedResultSerialNum = 120000
updatedAt = ...
```

恢复流程：

```text
1. 读取本模块 lastConsumedResultSerialNum。
2. 从 Result Replay Service 拉取 resultSerialNum > lastConsumedResultSerialNum。
3. 按顺序重放。
4. 每条事件幂等处理。
5. 成功后推进 checkpoint。
```

如果发现缺口：

```text
expected = 120001
actual = 120005
```

必须停止应用后续事件，先 replay 缺失事件或重新构建状态。

## 13. 背压与隔离要求

生产中不能让慢下游拖垮撮合。

要求：

```text
撮合主线程不等待清算、行情、OMS 消费完成
可靠结果日志优先于实时广播
实时广播可丢，但必须可 replay
清算和账本事件不可丢，必须从可靠日志恢复
慢消费者要隔离或断开
```

当前项目已删除 matching 内的结果广播组件。后续背压、丢弃、限速和重连处理应在独立 `result-publisher` 服务内实现，并且必须补上 replay 能力。

## 14. 监控和校验

生产一致性需要持续观测。

关键指标：

```text
lastProcessedRequestSerialNum
lastResultSerialNum
result archive max serialNum
result publisher lag
consumer lag by module
snapshot age
snapshot duration
snapshot checksum failure count
replay duration
serial gap count
duplicate command count
duplicate result consume count
```

关键校验：

```text
订单簿 checksum
resultSerialNum 连续性
requestSerialNum 连续性
archive recording position
下游 checkpoint 是否落后
清算账和成交事件是否一致
```

## 15. 分阶段改进计划

### 阶段一：保持当前全局序号模型

目标是把当前撮合恢复能力补完整。

建议：

```text
继续使用全局 sequencerId / serialNum
继续使用 resultSerialNum 作为清算幂等 sourceId
使用 ResultLogWriter / ArchiveResultLogWriter 作为 matching 内唯一结果写入边界
补下游 consumer checkpoint
补 result replay 查询能力
```

### 阶段二：抽离 Result Publisher

目标是隔离撮合状态机和下游推送。

建议：

```text
撮合线程只写 result log（已完成）
matching 内 MDC 广播代码删除（已完成）
Result Publisher 独立进程读取 result log
Publisher 按 resultSerialNum 推送下游
下游保存消费位点
支持从任意 resultSerialNum replay
```

### 阶段三：增强业务幂等

目标是增强重试、补偿和审计能力。

建议：

```text
协议预留 commandId
AMEND / CANCEL 增加 expectedOrderVersion
MATCH 事件增加 matchId / tradeId
控制面命令增加 controlSeq / configVersion
账本统一使用 sourceType + sourceId 幂等模型
```

### 阶段四：强化快照与日志边界

目标是让恢复边界可验证。

建议：

```text
快照记录 commandLogPosition
快照记录 resultLogPosition
快照记录 configVersion
启动时校验 result log 和 snapshot 边界
恢复后检查 resultSerialNum 不回退、不跳号
```

### 阶段五：面向多撮合分片升级

当前不需要立即做，但接口可以预留。

未来升级：

```text
globalCommandId
partitionId
partitionSeq
symbolSeq
per-partition snapshot
per-partition result stream
```

## 16. 当前推荐结论

对当前项目，推荐保持务实：

```text
1. 继续使用全局 serialNum / sequencerId。
2. 继续用 resultSerialNum 作为当前清算幂等 sourceId。
3. 不强制立即引入 matchId，但 MATCH 事件应预留字段。
4. PLACE 可以继续用 orderId 幂等。
5. AMEND / CANCEL 后续建议引入 commandId 和 expectedOrderVersion。
6. 优先补 Result Replay Service 和下游 checkpoint。
7. 优先抽离 Result Publisher，避免广播影响撮合主线程。
```

一句话总结：

```text
当前项目已经解决了“撮合状态机如何恢复”的核心问题；
生产化还要继续解决“撮合结果如何可靠发布、下游如何重放、清算如何幂等、快照与日志如何证明处在同一一致性边界”。
```

## 17. 后续待执行任务清单

以下任务基于当前已确认边界：

```text
matching 只写 Archive result-data-stream
matching 不再内置 MDC 广播
matching 不再提供 replay 查询
matching 不维护 resultSerialNum -> position 索引
索引和 replay 查询由后续独立 result-publisher / replay-service 异步构建
```

核心原则：

```text
Archive result-data-stream 是唯一事实源
replay-service 的数据库是派生读模型
派生索引可以删除后从 Archive result-data-stream 重建
```

### 17.1 文档任务边界更新

需要先把后续任务统一到新边界：

```text
旧方案：matching 写 result log + index log，等待两次 Archive 追进度
当前方案：matching 只写 result-data-stream，append 路径不逐条等待 Archive 录制追上
快照边界：同步快照前等待 Archive recordingPosition >= lastOfferedPosition
```

职责划分：

```text
matching
  -> 写 result-data-stream
  -> 维护 lastOfferedResultSerialNum / lastOfferedPosition
  -> append 路径不逐条等待 recordingPosition
  -> 同步快照前等待 Archive recordingPosition >= lastOfferedPosition
  -> 快照写入 maxResultSerialNum，恢复时只用 snapshot.maxResultSerialNum + 1

result-publisher / replay-service
  -> 消费 result-data-stream
  -> 解码 MatchResult
  -> 构建 resultSerialNum -> position 索引
  -> 保存 index / payload 到自己的数据库
  -> 提供 replay API
  -> 推送实时结果
```

### 17.2 Archive durable commit 强化（当前同步快照方案已落地）

当前 `ArchiveResultLogWriter.append()` 只负责把 result 按顺序 offer 到 Archive publication，并记录本进程已 offer 的最大边界。

当前已落地：

```text
append:
  逐条 offer MatchResult
  校验 resultSerialNum 严格递增
  更新 lastOfferedResultSerialNum
  更新 lastOfferedPosition
  不逐条等待 Archive recordingPosition

同步快照:
  读取 eventsHelper 当前 maxResultSerialNum
  校验 maxResultSerialNum == lastOfferedResultSerialNum（本进程本轮已有 offer 时）
  等待 Archive recordingPosition >= lastOfferedPosition
  等待成功后写入 snapshot.maxResultSerialNum

恢复:
  只使用 snapshot.maxResultSerialNum + 1 恢复 nextResultSerialNum
  不再使用 committedResultSerialNum 或 ResultRepository reconcile
```

目标是避免：

```text
快照声明的 maxResultSerialNum 已经进入撮合状态
但 Archive result-data-stream 尚未录制到该结果对应 position
```

这一步只在同步快照路径等待 result-data-stream 一次 Archive 追进度，不写第二条 index stream，也不在撮合 append 路径引入逐条持久化等待。

当前配置：

```properties
laser.matching.result-log.commit-timeout-ms=3000
```

后续还可继续补充指标和更细粒度的异常分类。

### 17.3 Result Archive Reader

在独立 `result-publisher / replay-service` 中实现 Archive 读取层。

职责：

```text
定位 result-data-stream recording
从指定 position replay
读取原始 SBE bytes
解码 SBE message header
识别 PlaceOrderResult / MatchOrderResult / CancelOrderResult / AmendOrderResult / OrderRejectResult 等
按 resultSerialNum 输出
发现重复、乱序、跳号时停止并告警
```

读取能力不放回 matching。

### 17.4 replay-service 派生索引

`replay-service` 消费 `result-data-stream` 后自行构建索引。

推荐索引字段：

```text
resultSerialNum
recordingId
startPosition
endPosition
templateId
requestSerialNum
createTime
```

要求：

- 索引构建必须幂等。
- 支持服务重启后从 checkpoint 继续扫描。
- 支持删除数据库后从 Archive 全量重建。
- 数据库不是事实源，只是查询加速读模型。

### 17.5 replay-service payload 存储策略

第一阶段建议保存：

```text
index + payload bytes
```

示例字段：

```text
resultSerialNum
templateId
requestSerialNum
createTime
payload bytes
```

这样 replay API 可以直接从数据库返回 payload，降低每次查询都访问 Archive 的复杂度。

长期语义保持不变：

```text
Archive 是事实源
数据库是可重建缓存
```

### 17.6 Result Replay API

由 `result-publisher / replay-service` 提供查询接口。

推荐接口：

```text
GET /result/replay?from=10001&to=10100
GET /result/replay?from=10001&limit=1000
GET /result/latest-serial-num
```

约束：

- 查询键必须基于 `resultSerialNum`。
- 返回结果必须严格递增。
- 单次返回必须限制最大条数。
- 下游发现缺口后统一从这里补齐，不访问 matching 状态机。
- 数据库缺失时可触发从 Archive 补扫，但不能返回不连续结果。

### 17.7 Result Publisher 实时推送

实时推送由 `result-publisher` 承担，推送源来自已消费并索引化的 result-data-stream。

职责：

```text
维护 lastPublishedResultSerialNum
按 resultSerialNum 顺序推送
处理 MDC / MQ / gRPC stream 等实时通道
处理背压、限速、重连、重复推送
```

matching 不再直接推送下游。

### 17.8 下游 consumer checkpoint

OMS、Clearing、MarketData、Audit 等消费方需要保存：

```text
consumerName + lastConsumedResultSerialNum
```

消费规则：

```text
N == last + 1  正常处理
N <= last      重复事件，幂等丢弃
N > last + 1   暂停推进，调用 replay 补齐缺口
```

checkpoint 成功落地后才能认为对应事件已被该消费者处理完成。

### 17.9 counter / mock 下游改造

当前 counter 的 `ResultMdcSubscriber` 仍是旧实时订阅模型。

后续需要改成：

```text
订阅独立 result-publisher
按 resultSerialNum 检测 gap
保存本地下游 checkpoint
发现缺口后调用 Result Replay Service 补齐
```

counter 只作为 mock 下游，不再直接订阅 matching。

### 17.10 快照与 result log 边界校验

快照中当前记录 `maxResultSerialNum`。

当前同步快照边界：

```text
snapshot.maxResultSerialNum == lastOfferedResultSerialNum
Archive recordingPosition >= lastOfferedPosition 后才允许写快照
恢复时 nextResultSerialNum = snapshot.maxResultSerialNum + 1
```

恢复语义：

```text
resultSerialNum 不回退
resultSerialNum 不跳号
最多因崩溃恢复和 raft log 重放产生重复 resultSerialNum
不会因快照越过 Archive 录制边界导致缺号
重复 resultSerialNum 由 replay-service 和下游消费者做幂等处理
```

后续如果引入异步快照，再重新设计并确认：

```text
snapshot 任务如何冻结 result 边界
是否需要在 snapshot 中记录 resultLogPosition / recordingPosition
异步快照期间新 result 如何与快照边界隔离
```
