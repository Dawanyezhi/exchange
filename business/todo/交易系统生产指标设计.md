# 交易系统生产指标设计

## 1. 目标

这份文档基于当前交易系统项目和 Day 24 的“抖动与运行时稳定性”内容，整理一套中心化交易所核心交易系统的生产指标。

指标目标不是只看系统是否“活着”，而是判断：

```text
订单是否及时进入系统
撮合是否稳定低延迟
结果事件是否可靠发布
清算和账本是否跟得上
行情是否连续、有序、低延迟
故障后是否能恢复和重放
```

生产指标应覆盖：

```text
延迟
吞吐
队列积压
错误率
连续性
资源使用
恢复能力
业务一致性
```

## 2. 指标分层

推荐按交易链路分层监控。

```text
Client / Gateway
  -> Session
  -> OMS
  -> Risk / Account Freeze
  -> Outbox Dispatcher
  -> Matching
  -> Result Log / Publisher
  -> Clearing
  -> Ledger / Account
  -> Market Data
```

每一层至少需要：

```text
请求量
成功量
失败量
拒绝量
耗时分位数
队列长度
积压时间
错误码分布
```

## 3. 全链路指标

| 指标 | 含义 | 说明 |
| --- | --- | --- |
| `order_e2e_latency` | 用户请求到最终订单回报耗时 | 区分 accepted、rejected、filled、canceled |
| `order_accept_latency` | 下单请求到订单 accepted 回报耗时 | 衡量接入、OMS、风控、投递链路 |
| `order_to_match_latency` | 订单通过风控到进入撮合耗时 | 衡量 Outbox/Dispatcher/Matching ingress |
| `match_to_result_publish_latency` | 撮合完成到结果发布耗时 | 衡量结果日志和 Publisher |
| `match_to_clearing_latency` | 撮合完成到清算消费耗时 | 衡量清算滞后 |
| `match_to_market_data_latency` | 撮合完成到行情发布耗时 | 衡量行情实时性 |

分位数建议：

```text
P50
P90
P99
P99.9
max
```

生产中不要只看平均值。平均值容易掩盖长尾问题。

## 17. 告警建议

### 17.1 严重告警

这些情况应立即告警：

```text
request serialNum 不连续
resultSerialNum 不连续
撮合服务 leader 异常切换
结果日志写入失败
Archive recording position 长时间不推进
清算消费停止
账本写入失败
账户出现负余额
快照 checksum 失败
```

### 17.2 延迟告警

这些情况应按阈值告警：

```text
下单 P99 超过目标
撮合 P99.9 超过目标
清算 lag 超过目标
行情发布延迟超过目标
Outbox 最老未 ACK 时间超过目标
Publisher lag 持续扩大
```

### 17.3 容量告警

这些情况说明系统接近过载：

```text
入口限流持续触发
Outbox 积压持续增长
Aeron backpressure 持续增长
清算队列持续增长
慢行情客户端大量增加
GC pause 明显上升
CPU run queue 持续升高
```

## 18. 指标目标如何定

没有真实业务规模和压测数据前，不建议硬写生产阈值。

推荐流程：

```text
1. 在准生产环境跑基准压测。
2. 记录空载、常态、峰值、极端压力下的 P50/P99/P99.9。
3. 根据业务目标确定 SLO。
4. 告警阈值设置为 SLO 的 70% / 90% / 100% 多级告警。
5. 上线后按真实流量持续修正。
```

示例模板：

| 指标 | 目标值 | 告警值 | 严重告警 |
| --- | --- | --- | --- |
| 下单 P99 | 待压测确定 | 目标 70% | 目标 100% |
| 撮合 P99 | 待压测确定 | 目标 70% | 目标 100% |
| 清算 lag | 待压测确定 | 目标 70% | 目标 100% |
| 行情延迟 P99 | 待压测确定 | 目标 70% | 目标 100% |
| Outbox oldest unacked age | 待压测确定 | 目标 70% | 目标 100% |

## 20. 一句话总结

生产指标不是单纯监控 CPU、内存和接口成功率。

交易系统必须重点监控：

```text
顺序是否连续
延迟是否稳定
队列是否积压
事件是否可重放
下游是否落后
账务是否一致
故障后是否能恢复
```

这些指标共同决定交易系统是否真正具备生产可用性。
