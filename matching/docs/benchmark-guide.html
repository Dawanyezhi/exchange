Laser 撮合引擎 Benchmark 使用手册

项目：laser-matching-engine  |  目标方法：MatchEngine#placeOrder(MatchOrder)  |  框架：JMH 1.37  |  Java：17

本手册面向希望了解和使用撮合引擎性能测试套件的开发者和学员。通过阅读本手册，您将掌握：

  • JMH 基准测试的核心概念和生命周期
  • 撮合引擎 placeOrder 方法的完整处理流程
  • 9 大 Benchmark 场景的设计思路和覆盖的代码路径
  • 如何配置、运行 Benchmark 并解读结果

什么是 JMH

JMH（Java Microbenchmark Harness）是 OpenJDK 官方提供的微基准测试框架。 它解决了手写 System.nanoTime() 计时的常见陷阱：

陷阱JMH 如何解决
JIT 编译器优化掉"无用"代码Blackhole 消费返回值，阻止死码消除
JVM 启动时的冷启动效应@Warmup 先预热，让 JIT 充分编译
JVM 状态残留影响结果@Fork 在独立 JVM 进程中运行
系统负载波动多次迭代取统计值（均值、标准差）

为什么需要 Benchmark 撮合引擎

撮合引擎是交易系统的核心，每笔订单的处理延迟直接影响用户体验和系统吞吐量。通过 Benchmark 我们可以：

  1. 量化性能基线 — 知道各种订单类型的处理耗时（微秒级）
  2. 发现性能瓶颈 — 对比不同场景，定位最耗时的路径
  3. 验证优化效果 — 修改代码后，对比前后的 Benchmark 数据
  4. 容量规划 — 根据吞吐量数据估算系统承载能力

二、模块架构

文件清单

Benchmark 模块位于 src/test/java/com/laser/exchange/matching/benchmark/ 包下，包含 4 个类：

BenchmarkConfig.java — 配置中心

管理所有可配置参数（预填充订单数、价格档位数、JMH 迭代次数等），支持通过 JVM -D 属性覆盖默认值。

// 从系统属性中读取配置，未设置则使用默认值
BenchmarkConfig config = BenchmarkConfig.fromSystemProperties();
int orders = config.getPrefillOrders();  // 默认 10000
OrderGenerator.java — 订单工厂

快速构造各类型的 MatchOrder，内置自增 ID 生成器保证唯一性。

// 构造一个 GTC 限价买单，价格 49000，数量 1.0
MatchOrder order = OrderGenerator.limitBuy(49000, 1.0, TimeInForceEnum.GTC);

// 构造一个带 STP 配置的卖单
MatchOrder stp = OrderGenerator.limitSellWithStp(49100, 1.0,
        TimeInForceEnum.GTC, accountId, stpAccountId, StpStrategyEnum.CANCEL_MAKER);
MatchEngineBenchmark.java — JMH 基准测试主类

包含 9 个 @Benchmark 方法，每个方法测试一个特定的撮合场景。通过 @Setup 注解管理测试状态。

MatchEngineBenchmarkRunner.java — 运行器

提供 main() 方法，读取配置并启动 JMH。运行结果输出到控制台和 benchmark-result.json。

三、JMH 生命周期

理解 JMH 的生命周期是正确编写和解读 Benchmark 的关键。本项目使用了两层 @Setup：

sequenceDiagram participant Runner as Runner.main() participant JMH as JMH Framework participant Fork as Forked JVM participant Setup1 as @Setup(Trial) participant Setup2 as @Setup(Invocation) participant BM as @Benchmark Runner->>JMH: 启动 Benchmark JMH->>Fork: Fork 新 JVM (隔离状态) Fork->>Setup1: 读取配置 (仅执行 1 次) Note over Setup1: prefillOrders = 10000
prefillPriceLevels = 100 loop 每次 Warmup / Measurement 迭代 loop 每次 Invocation Fork->>Setup2: 创建引擎 + 预填充订单簿 Note over Setup2: new MatchEngine()
prefillOrderBook() Fork->>BM: 执行 @Benchmark (仅测这里) Note over BM: engine.placeOrder(order) BM-->>Fork: 返回结果 end end Fork-->>JMH: 汇总统计数据 JMH-->>Runner: 输出结果
关键设计：预填充订单簿的操作在 @Setup(Level.Invocation) 中完成， 不在 @Benchmark 方法内。这意味着 JMH 不会将预填充时间计入测量结果， 只测量真正的 placeOrder 调用。
注解执行时机本项目用途
@Setup(Level.Trial)整个 Benchmark 开始前执行 1 次读取 BenchmarkConfig 配置参数
@Setup(Level.Invocation)每次 @Benchmark 方法调用前执行创建新的 MatchEngine + 预填充订单簿
@Benchmark被 JMH 反复调用并计时只执行一笔 placeOrder，纯净测量

五、9 大测试场景

每个场景覆盖 placeOrder 的不同代码路径，从最轻量到最重量排列：

1 GTC 不交叉挂单 O(log N)

买单价格远低于卖盘最优价，不触发撮合，直接加入 TreeMap 订单簿。这是撮合引擎最快的路径。

方法名scenario1_GTC_NoMatch_AddToBook
代码路径placeOrder → processGtcOrder → gtcNotCross()=true → addOrder()
复杂度O(log N)，N 为当前价格档位数
真实场景做市商铺单、用户挂远离盘口的限价单
flowchart LR A["placeOrder"] --> B["processGtcOrder"] --> C{"gtcNotCross?"} -- "true" --> D["addOrder O(logN)"] style D fill:#198754,color:#fff 
@Benchmark
public void scenario1_GTC_NoMatch_AddToBook(Blackhole bh) {
    MatchOrder order = OrderGenerator.limitBuy(BASE_PRICE - 500, DEFAULT_QTY, TimeInForceEnum.GTC);
    engine.placeOrder(order);
    bh.consume(order);
}
2 GTC 完全成交（单档位） O(M)

买单价格等于卖盘最优价，与卖盘一笔订单完全成交。这是最基本的 taker 撮合路径。

方法名scenario2_GTC_SingleMatch
代码路径placeOrder → processGtcOrder → isCross=true → tryMatchInstantly → match → doMatch
复杂度O(1) 成交 + O(log N) 清理
真实场景普通用户吃单
3 GTC 多档位成交 O(N*M)

大单横扫 10 个价格档位，测量跨档位遍历撮合的性能开销。

方法名scenario3_GTC_MultiLevelMatch
代码路径placeOrder → processGtcOrder → tryMatchInstantly → [10 次 match] → doMatch(多次)
复杂度O(N*M)，N=10 档位，M=每档位订单数
真实场景大单扫盘、极端行情
4 IOC 立即成交 O(M)

IOC 订单立即撮合，剩余部分直接撤销（不挂单）。与 GTC 共享 tryMatchInstantly 路径。

方法名scenario4_IOC_InstantMatch
代码路径placeOrder → processIocOrder → isCross=true → tryMatchInstantly → cancel 剩余
复杂度与 GTC 单档位成交接近
真实场景要求立即成交的交易策略
5 FOK 完全成交（两段式撮合） 2x O(N*M)

FOK 采用两段式撮合：先模拟遍历（Phase1）验证是否能完全成交，确认后才执行真实成交（Phase2）。

方法名scenario5_FOK_FullFill
代码路径placeOrder → processFokOrder → FokOrderProcessor.process → Phase1 → Phase2
复杂度约为 GTC 同等成交量的 2 倍（双重遍历）
真实场景要求全量成交或完全放弃的确定性交易
flowchart LR A["FOK placeOrder"] --> B["Phase1 预撮合
(深拷贝+模拟)"] B --> C{"能完全成交?"} C -- "是" --> D["Phase2 真实撮合"] C -- "否" --> E["cancel 全部撤单"] style D fill:#198754,color:#fff style E fill:#dc3545,color:#fff
6 FOK 撤单（预撮合失败） O(N*M)

FOK 数量超过对手盘总量，预撮合发现无法完全成交，直接撤单。只执行 Phase1，不执行 Phase2。

方法名scenario6_FOK_Cancel
代码路径placeOrder → processFokOrder → Phase1(遍历全部对手盘，判定失败) → cancel
复杂度约为 FOK 完全成交的 50%（只有 Phase1）
真实场景流动性不足时的 FOK 订单
7 POST_ONLY 挂单 O(log N)

POST_ONLY 订单只做 maker。价格不交叉时直接挂入订单簿，是最轻量的处理路径。

方法名scenario7_PostOnly_AddToBook
代码路径placeOrder → processPostOnlyOrder → isCross=false → addOrder
复杂度O(log N)
真实场景做市商批量铺单
8 深度订单簿扫盘（压力测试） O(N*M)

模拟极端行情：一笔大单扫光整个卖盘。这是撮合引擎的最坏情况。

方法名scenario8_DeepBook_FullSweep
代码路径placeOrder → processGtcOrder → tryMatchInstantly → [遍历全部档位] → match(全量)
复杂度O(N*M)，N=全部价格档位数，M=每档位订单数
真实场景市场崩盘、巨量市价单
9 混合场景（综合吞吐量） 综合

随机生成 4 种订单类型，模拟真实交易环境的混合工作负载。

方法名scenario9_MixedWorkload
订单分布GTC 挂单 25% | GTC 成交 25% | IOC 成交 25% | POST_ONLY 25%
复杂度各场景加权平均
真实场景正常交易时段的综合负载

所有参数均通过 JVM 系统属性 (-D) 配置，未设置时使用默认值。

七、运行指南

1 IDE 直接运行（推荐新手）

  1. 在 IDE 中打开 MatchEngineBenchmarkRunner.java
  2. 右键 → Run main() 方法
  3. （可选）在 VM Options 中添加 -D 参数： 
    -Dbenchmark.prefill.orders=5000 -Dbenchmark.prefill.priceLevels=50

2 Maven 命令行运行

# 使用默认配置运行
mvn test-compile exec:java \
    -Dexec.mainClass="com.laser.exchange.matching.benchmark.MatchEngineBenchmarkRunner" \
    -Dexec.classpathScope="test"

# 自定义预填充订单数和价格档位数
mvn test-compile exec:java \
    -Dexec.mainClass="com.laser.exchange.matching.benchmark.MatchEngineBenchmarkRunner" \
    -Dexec.classpathScope="test" \
    -Dbenchmark.prefill.orders=5000 \
    -Dbenchmark.prefill.priceLevels=50
运行输出
  • 控制台：实时输出每个场景的 Throughput (ops/us) 和 Average Time (us/op)
  • JSON 文件：benchmark-result.json，可导入 JMH Visualizer 可视化分析

十、常见问题

可能原因：prefillOrders 设置过大（如 100000），每次 invocation 都要预填充大量订单。

解决方案：减小 -Dbenchmark.prefill.orders 的值（建议 1000~10000），或减少 warmupIterations 和 measurementIterations。

撮合引擎的特殊性：每次 placeOrder 都会修改订单簿状态（成交、移除订单等）。如果使用 Level.Iteration，第一次调用后订单簿已被修改，后续调用的测量结果不准确。

Level.Invocation 确保每次测量都在干净的订单簿上进行，结果更可信。

可以，但不推荐。forks=0 在当前 JVM 中运行，JIT 编译状态、GC 堆布局都可能影响结果。仅建议在调试 Benchmark 代码本身时使用 forks=0，正式评估请使用 forks≥1。

修改 MatchEngineBenchmarkRunner 中的 .include() 参数：

// 只运行场景 1
.include("MatchEngineBenchmark.scenario1")

// 运行场景 1 和场景 2
.include("MatchEngineBenchmark.scenario[12]")

include 接受正则表达式，可以灵活匹配方法名。
