1. 概述

什么是 MatchEngineStressTest

MatchEngineStressTest 是 Laser Matching Engine 专属的自定义压力测试框架。它采用基于时间的持续运行模式，对撮合引擎在各类典型交易场景下的吞吐量和延迟进行稳态测量。框架完全使用纯 Java 实现，零外部依赖，可通过 main 方法直接运行。

核心特性：
  • 基于时间的持续测量 -- 模拟真实生产环境的稳态负载
  • 多轮测量 + 标准差 -- 量化结果稳定性
  • TP50/90/99/99.9 百分位延迟 -- 精细刻画尾部延迟
  • ASCII 直方图 -- 直观展示延迟分布
  • 9 个覆盖全面的测试场景 -- 从单档成交到深度扫盘

与 JMH Benchmark 的区别

维度 JMH Benchmark MatchEngineStressTest
目的 微观延迟精度 宏观吞吐 + 百分位延迟
百分位指标 仅 avg TP50 / TP90 / TP99 / TP99.9
稳定性验证 多 fork 隔离 多轮持续运行 + 标准差

3. 测量策略

本节是理解压力测试结果的关键。所有场景使用统一的测量方法：周期性重建引擎 + 纯操作计时，确保结果直接可比。

3.1 总体流程

每个场景的测量遵循「预热 + 多轮测量 + 汇总」三阶段流程：

3.2 统一测量流程

所有场景均使用统一的 runTimed() 方法。每个场景配置了 opsPerEngine 参数，当操作次数达到该阈值时自动重建引擎。引擎重建时间不计入吞吐量测量，只累计纯操作延迟。

3.3 吞吐量计算公式

统一公式（适用于所有场景）
throughput = totalOps * 1e9 / sumOfOperationLatencies

其中 sumOfOperationLatencies 是所有操作的纳秒延迟之和。排除了：

  • 引擎重建时间（createEngine + prefillOrderBook）
  • 批次间的控制流开销

测量的是「纯操作吞吐量」，体现引擎自身性能上限。所有场景使用相同的计算方式，结果直接可比。

统一测量方法的优势：
所有场景均采用周期性引擎重建 + 纯操作计时的方式，消除了过去非破坏性场景与破坏性场景之间因计算方式不同（挂钟时间 vs 操作延迟之和）导致的数据不可比问题。现在 9 个场景的吞吐量数值具有完全一致的统计口径，可以直接横向对比。

3.4 Deadline 检查优化

每次操作都调用 System.nanoTime() 来检查是否超时会带来额外开销（约 80ns/次）。框架采用了每 256 次操作检查一次的策略：

// 常量定义
private static final int DEADLINE_CHECK_INTERVAL = 256;
private static final int DEADLINE_CHECK_MASK = DEADLINE_CHECK_INTERVAL - 1;  // 0xFF

// 热循环中使用位运算代替取模
if ((totalOps & DEADLINE_CHECK_MASK) == 0) {
    if (System.nanoTime() >= deadline) break;
}

使用 2 的幂（256 = 28）和位掩码（& 0xFF），可以将取模运算替换为一次位与运算，在热循环中比 % 256 更快。

3.5 延迟采样策略

每轮测量最多采集 1,000,000 个延迟样本（MAX_SAMPLES_PER_ROUND），多轮之间通过 LatencyStats.merge() 合并。

  • 内存占用：1M 样本 x 8 字节 = 8 MB/轮
  • 合并后排序计算百分位：Arrays.sort()，复杂度 O(N log N)
  • 百分位定位使用 ceiling 方式：idx = ceil(p/100 * N) - 1

场景 1: GTC 不交叉挂单 opsPerEngine = 5,000

描述：以远低于最优卖价的价格（BASE_PRICE - 500）提交 GTC 限价买单。由于价格不交叉，订单直接挂入买方订单簿，不发生撮合。

代码路径：MatchEngine.placeOrder() → processGtcOrder() → addOrder()

测试目标：衡量订单簿的纯插入性能，订单簿使用 TreeMap 存储，插入复杂度与价格档位数相关。

时间复杂度O(log N)

场景 2: GTC 单档成交 opsPerEngine = 5,000

描述：以卖方最优价格（askBestPrice）提交 GTC 限价买单，每次消耗一个卖单。当卖方最优档订单耗尽时需要重建引擎。

代码路径：MatchEngine.placeOrder() → processGtcOrder() → matchOrder() → 单笔成交

测试目标：衡量单档位最优价撮合的性能，这是交易所最常见的成交场景。

时间复杂度O(1)
opsPerEngine5,000 (totalAskOrders)
TIFGTC
场景 3: GTC 多档位成交 opsPerEngine = 10

描述：以横跨 10 个价格档位的数量和价格提交 GTC 限价买单，一次操作吃掉 10 个档位的全部挂单。每次操作消耗 10 个档位，priceLevels / 10 = 10 次操作后引擎重建。

代码路径：MatchEngine.placeOrder() → processGtcOrder() → matchOrder() → 遍历多档位多笔成交

测试目标：衡量跨多档位撮合的性能，每次操作需要遍历 N 个档位 x M 笔订单。

时间复杂度O(N*M)
opsPerEngine10 (priceLevels/10)
TIFGTC
场景 4: IOC 立即成交 opsPerEngine = 5,000

描述：以卖方最优价格提交 IOC（Immediate-Or-Cancel）限价买单。IOC 订单立即与最优价成交，未成交部分自动取消，不会挂入订单簿。

代码路径：MatchEngine.placeOrder() → processIocOrder() → matchOrder() → 单笔成交 + 取消剩余

测试目标：衡量 IOC 订单的处理性能。由于省去了挂单步骤，理论上比 GTC 成交更快。

时间复杂度O(1)
opsPerEngine5,000 (totalAskOrders)
TIFIOC
场景 5: FOK 完全成交 opsPerEngine = 5,000

描述：以卖方最优价格提交 FOK（Fill-Or-Kill）限价买单。FOK 订单需要先进行预撮合（prematch）检查是否有足够数量可以完全成交，然后一次性全部成交或全部取消。

代码路径：MatchEngine.placeOrder() → processFokOrder() → prematch检查 → matchOrder() → 单笔成交

测试目标：衡量 FOK 订单的处理性能，包含额外的预撮合检查开销。

时间复杂度O(1) + prematch
opsPerEngine5,000 (totalAskOrders)
TIFFOK
场景 6: FOK 撤单 opsPerEngine = 5,000

描述：以超出订单簿所有卖单的数量（impossibleQty = prefillOrders * 10）提交 FOK 买单。预撮合检查必定失败，订单被直接拒绝，不消耗订单簿。

代码路径：MatchEngine.placeOrder() → processFokOrder() → prematch检查(遍历全部档位) → 数量不足 → 撤单

测试目标：衡量 FOK 预撮合在最坏情况下（需要遍历所有档位后才发现数量不足）的性能。

时间复杂度O(N*M)
opsPerEngine5,000 (totalAskOrders)
TIFFOK
场景 7: POST_ONLY 挂单 opsPerEngine = 5,000

描述：以远低于最优卖价的价格（BASE_PRICE - 500）提交 POST_ONLY 限价买单。POST_ONLY 订单保证只挂单不吃单，检查通过后直接挂入订单簿。

代码路径：MatchEngine.placeOrder() → processPostOnlyOrder() → 交叉检查 → addOrder()

测试目标：衡量 POST_ONLY 订单类型的挂单性能，与 GTC 不交叉挂单类似但多了 POST_ONLY 专用的交叉检查。

时间复杂度O(log N)
opsPerEngine5,000 (totalAskOrders)
TIFPOST_ONLY
场景 8: 深度订单簿扫盘 opsPerEngine = 1

描述：以覆盖所有卖方档位的价格和数量提交 GTC 买单，一次操作清空整个卖方订单簿（5,000 笔卖单）。这是最极端的撮合场景。

代码路径：MatchEngine.placeOrder() → processGtcOrder() → matchOrder() → 遍历 100 档位 x 50 订单/档

测试目标：衡量全深度扫盘的极端性能，单次操作处理量最大。

时间复杂度O(N*M)
opsPerEngine1（一次清空）
TIFGTC
场景 9: 混合场景 opsPerEngine = 2,500

描述：随机混合 4 种订单类型：GTC 不交叉挂单（25%）、GTC 成交（25%）、IOC 成交（25%）、POST_ONLY 挂单（25%）。模拟真实交易环境中的混合订单流。

代码路径：随机调用以上 4 种订单类型对应的处理路径

测试目标：衡量引擎在混合负载下的综合性能。由于约 50% 的订单会成交，属于破坏性场景。

时间复杂度随机

6. 配置参数

所有参数通过 Java 系统属性（-D 参数）传入，未指定时使用默认值。

参数名 默认值 说明 影响
stress.prefill.orders 10,000 预填充订单簿的总订单数（买卖各一半） 影响订单簿深度，买方 5,000 + 卖方 5,000
stress.prefill.priceLevels 100 预填充的价格档位数 影响每个档位的订单密度（50 笔/档），影响 O(log N) 和 O(N*M) 场景的性能

8. 结果解读

8.1 吞吐量数值的含义

吞吐量以 ops/s（每秒操作数）表示。一次操作 = 一次 engine.placeOrder(order) 调用。

所有场景均使用统一公式计算：throughput = totalOps x 109 / sumOfOperationLatencies。该公式仅累计每次 placeOrder 的纯操作耗时，排除了引擎重建、预填充等非操作开销，体现的是「纯操作吞吐量」，即撮合引擎自身的处理能力上限。

8.2 百分位延迟（TP Percentiles）

指标含义关注点
TP50 50% 的操作延迟低于此值 中位数延迟，反映典型性能
TP90 90% 的操作延迟低于此值 绝大部分请求的延迟上限
TP99 99% 的操作延迟低于此值 尾部延迟，对交易系统至关重要
TP99.9 99.9% 的操作延迟低于此值 极端尾部，通常受 GC、OS 调度影响

百分位计算方法：将所有延迟样本排序，取 ceil(p/100 * N) - 1 位置的值。样本来源于所有测量轮次的合并。

8.3 误差百分比（Error%）

误差 = 标准差 / 均值 x 100%，反映多轮测量结果的离散程度。

< 5%
结果稳定可信
5% - 20%
存在波动，建议增加轮次
> 20%
结果不稳定，需排查干扰

8.4 测量开销说明

每次操作需要调用 System.nanoTime() 来记录延迟，该调用本身有约 ~25-80ns 的开销。对于延迟极低的操作（如场景 2、4、5 的 100-130ns 级别），测量开销占比可达 30-60%。

实际操作延迟比测量值更低。测量值是操作延迟 + nanoTime 开销的总和。这是所有纳秒级基准测试的固有限制，不影响场景间的相对比较。

8.5 为何 GTC 单档成交吞吐量高于 GTC 不交叉挂单？

在统一测量方法下，场景 2（GTC 单档成交，791 万 ops/s）的吞吐量高于场景 1（GTC 不交叉挂单，745 万 ops/s）。这不是测量误差，而是由代码路径的本质差异决定的：

  1. O(1) vs O(log N) 的复杂度差异：单档成交使用 TreeMap.firstEntry() 获取最优价 + LinkedList.poll() 取出第一笔订单，均为 O(1) 操作；而不交叉挂单需要 TreeMap.computeIfAbsent() 进行订单簿插入，这是 O(log N) 操作。
  2. 成交路径不修改买方 TreeMap：单档成交的买单与卖单完全成交（fully filled），买单不会被插入买方订单簿的 TreeMap，省去了一次 O(log N) 插入。
  3. 无需创建 DepthLine 对象：不交叉挂单路径需要为新的价格档位创建 DepthLine 对象并维护订单链表；成交路径直接从已有的档位中取出订单，不创建新对象。
  4. 挂单路径有额外日志开销：不交叉挂单路径包含 log.debug() 日志调用（即使日志级别未开启，方法调用本身仍有开销）；成交路径不含此类调用。

总结：在最优价匹配的场景中，撮合操作比订单簿插入更轻量，这是撮合引擎数据结构设计的自然结果。

9. 常见问题

Q1: 场景 1 和场景 7 的误差是否可信？

在统一测量方法下，场景 1（GTC 不交叉挂单）的误差为 ±1.3%，场景 7（POST_ONLY 挂单）的误差为 ±0.3%，均处于非常稳定的水平。这是因为所有场景现在都采用周期性引擎重建（opsPerEngine=5,000），每 5,000 次操作后重建引擎，订单簿不会无限膨胀。过去这两个场景误差较高（6.9%、9.7%），正是因为引擎持续复用导致订单簿不断增大，O(log N) 的 N 值随时间增长，各轮次性能逐渐下降。统一测量方法从根本上消除了这个问题。

Q2: 场景 6（FOK 撤单）为什么这么慢？

FOK 撤单场景使用了一个不可能满足的数量（prefillOrders * 10 = 100,000），迫使 FOK 预撮合逻辑遍历整个卖方订单簿的所有 100 个价格档位 x 50 笔订单/档 = 5,000 笔订单后，才能确认数量不足并拒绝。这是 O(N*M) 的最坏情况遍历，每次操作约 60us。

Q3: 如何增大订单簿规模来测试更真实的场景？

通过系统属性调整预填充参数：

-Dstress.prefill.orders=100000 -Dstress.prefill.priceLevels=1000

这将创建 50,000 买单 + 50,000 卖单，分布在 1,000 个价格档位上（50 笔/档）。注意，更大的订单簿会显著影响 O(log N) 和 O(N*M) 场景的性能数值。

Q4: 测量结果受系统负载影响大吗？

是的。建议在运行压力测试时：

  • 关闭不必要的后台应用（特别是浏览器、IDE 的代码分析）
  • 确保笔记本电脑接通电源（避免省电模式降频）
  • 使用默认或深度模式（更多轮次 = 更稳定的均值）
  • 关注误差百分比：<5% 说明结果可信
Q5: 为什么不直接使用 JMH？

JMH 是优秀的微基准测试框架，本项目同时提供了 JMH Benchmark。但 JMH 的默认输出只有平均延迟，不提供 TP50/90/99/99.9 百分位分布。此外，JMH 对破坏性场景（需要反复重建订单簿）的支持需要较多样板代码（@Setup(Level.Iteration) + @State），而 MatchEngineStressTest 内置了自动引擎重建和时间排除逻辑，更加简洁。

Q6: 延迟采样的 1M 上限会影响百分位精度吗？

对于大多数场景，1M 样本足够精确。以场景 4（IOC 立即成交）为例，3 轮测量共产生约 2800 万次操作，每轮采样上限 1M，合并后约 3M 样本。TP99.9 = 第 2,997,000 个样本的值，精度足够。只有当操作延迟的分布极端不均匀时，采样上限才可能略微影响最高百分位的精度。

Q7: 为什么所有场景都需要周期性重建引擎？

统一测量方法要求所有场景使用相同的计算公式（totalOps x 1e9 / sumOfLatencies）。过去，非破坏性场景使用挂钟时间计算吞吐量，破坏性场景使用操作延迟之和，两者的统计口径不同，导致跨场景的吞吐量数值不可直接比较。通过为所有场景设置 opsPerEngine（即周期性重建引擎），统一了测量方式：所有场景都在固定大小的订单簿上运行，所有场景都使用纯操作计时。这使得 9 个场景的吞吐量具有完全一致的含义，可以直接横向对比，消除了方法论差异带来的数据失真。
