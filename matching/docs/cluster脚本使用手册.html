Aeron Cluster 操作手册

Laser Matching Engine  |  cluster.sh 使用指南

文档说明

本文档是 cluster.sh 脚本的操作手册，用于管理 Laser Matching Engine 的 Aeron Cluster 集群生命周期， 包括编译、启动、停止、状态查看和日志查看。

脚本位置：项目根目录下的 cluster.sh，使用前请确保有执行权限：chmod +x cluster.sh
命令总览
命令说明
./cluster.sh help显示帮助信息
./cluster.sh build仅编译打包
./cluster.sh start <nodeId>编译并启动指定节点（三节点集群模式）
./cluster.sh start-all编译并启动全部 3 个节点
./cluster.sh start-single编译并以单节点模式启动，直接成为 Leader
./cluster.sh stop <nodeId>停止指定节点
./cluster.sh stop-all停止全部节点
./cluster.sh restart <nodeId>重启指定节点（stop + build + start）
./cluster.sh status查看各节点运行状态
./cluster.sh log <nodeId>实时查看指定节点日志

前置条件

JDK17+
Maven3.9+
操作系统macOS / Linux
权限chmod +x cluster.sh

目录结构

脚本运行时会在 /tmp/laser/ 下自动创建以下目录：

/tmp/laser/
├── pids/                  # PID 与角色文件（自动创建）
│   ├── node-0.pid
│   ├── node-0.role        # 当前角色（LEADER/FOLLOWER/CANDIDATE）
│   ├── node-1.pid
│   ├── node-1.role
│   ├── node-2.pid
│   └── node-2.role
├── logs/                  # 日志文件（自动创建）
│   ├── node-0.log
│   ├── node-1.log
│   └── node-2.log
└── node-0/                # Aeron 集群数据目录（由应用创建）
    ├── aeron/
    ├── archive/
    ├── consensus/
    └── service/
/tmp 目录在系统重启后可能被清理。生产环境请修改脚本中的 PID_DIR 和 LOG_DIR 路径。

help — 显示帮助

./cluster.sh help

打印所有可用命令、示例和集群模式说明。无参数时也默认执行 help。

build — 编译打包

./cluster.sh build

执行 mvn clean package -DskipTests -q，生成可执行 JAR：

target/laser-matching-engine-1.0.0-SNAPSHOT.jar

所有 start 系列命令会自动先执行 build，无需手动调用。

start — 启动指定节点

./cluster.sh start <nodeId>
参数说明取值
nodeId节点编号0、1、2
执行流程
flowchart LR A["mvn package"] --> B["检查是否已运行"] B -->|未运行| C["java -jar ... &"] C --> D["写入 PID 文件"] B -->|已运行| E["跳过启动"] style C fill:#198754,color:#fff,stroke:none style E fill:#ffc107,stroke:none
示例
# 编译并启动节点 0
./cluster.sh start 0

# 编译并启动节点 1
./cluster.sh start 1

此命令用于三节点集群模式，默认 clusterNodeSize=3。 单独启动一个节点时，该节点会等待其他节点上线后进行 Raft 选举。

start-all — 启动全部节点

./cluster.sh start-all

编译后依次启动节点 0、1、2，每个节点间隔 2 秒（避免端口冲突）。

flowchart LR A["mvn package"] --> B["start node-0"] B -->|2s| C["start node-1"] C -->|2s| D["start node-2"] D --> E["3 节点集群就绪
Raft 选举 Leader"] style E fill:#198754,color:#fff,stroke:none

start-single — 单节点集群

./cluster.sh start-single

以单节点集群模式启动。固定 nodeId=0，通过 JVM 参数覆盖 clusterNodeSize=1， 使该节点成为集群唯一成员，立即选举为 Leader，无需等待其他节点。

与三节点模式的区别
特性三节点集群单节点集群
节点数量3（nodeId: 0, 1, 2）1（nodeId: 0）
Leader 选举Raft 共识，需多数节点在线立即成为 Leader
高可用容忍 1 个节点故障无容错能力
适用场景生产环境开发调试、功能验证
实现原理
# 脚本实际执行的命令
java --add-opens java.base/jdk.internal.misc=ALL-UNNAMED \
     --add-opens java.base/sun.nio.ch=ALL-UNNAMED \
     --add-opens java.base/java.nio=ALL-UNNAMED \
     --add-opens java.base/java.lang=ALL-UNNAMED \
     -Dlaser.aeron.cluster.config.nodeId=0 \
     -Dlaser.aeron.cluster.config.clusterNodeSize=1 \
     -jar target/laser-matching-engine-1.0.0-SNAPSHOT.jar

关键参数 -Dlaser.aeron.cluster.config.clusterNodeSize=1 覆盖 application.properties 中的默认值 3，使 buildClusterMembers() 仅生成一个成员条目，Aeron Raft 协议自动将唯一成员选举为 Leader。

stop — 停止指定节点

./cluster.sh stop <nodeId>
停止流程
flowchart LR A["读取 PID 文件"] --> B["发送 SIGTERM"] B --> C{"等待退出
最多 10 秒"} C -->|进程已退出| D["清理 PID 文件"] C -->|超时未退出| E["发送 SIGKILL"] E --> D style B fill:#ffc107,stroke:none style E fill:#dc3545,color:#fff,stroke:none style D fill:#198754,color:#fff,stroke:none
  • SIGTERM：触发 JVM Shutdown Hook → Spring @PreDestroy → 调用 CloseHelper.quietClose() 优雅关闭 Aeron 组件
  • SIGKILL：10 秒超时后强制终止，作为兜底保障
示例
./cluster.sh stop 0    # 停止节点 0
./cluster.sh stop 1    # 停止节点 1

stop-all — 停止全部节点

./cluster.sh stop-all

依次停止节点 0、1、2。每个节点的停止逻辑与 stop 命令相同。

restart — 重启指定节点

./cluster.sh restart <nodeId>
参数说明取值
nodeId节点编号0、1、2
执行流程
flowchart LR A["stop 节点"] --> B["mvn package"] B --> C["start 节点"] C --> D["重启完成"] style A fill:#dc3545,color:#fff,stroke:none style B fill:#ffc107,stroke:none style C fill:#198754,color:#fff,stroke:none

组合 stop + build + start 三个步骤，适用于代码修改后需要重新部署单个节点的场景。

示例
# 重启节点 0（先停止，再编译，再启动）
./cluster.sh restart 0

# 重启节点 2
./cluster.sh restart 2
提示：若节点未在运行中，restart 会跳过停止步骤，直接编译并启动。

status — 查看运行状态

./cluster.sh status

输出各节点运行状态表（含集群角色）：

  NodeId     PID        HTTP Port  Role         Status
  ------     ---        ---------  ----         ------
  0          12345      8080       LEADER       running
  1          12346      8081       FOLLOWER     running
  2          -          8082       -            stopped
Role 列说明
角色说明
LEADER领导者，处理客户端请求
FOLLOWER跟随者，复制 Leader 日志
CANDIDATE候选人，正在发起选举
-节点未运行或角色信息不可用

角色信息由 MatchEngineClusteredService 在 onStart 和 onRoleChange 回调时写入 /tmp/laser/pids/node-{id}.role 文件，节点停止时自动清理。

log — 实时查看日志

./cluster.sh log <nodeId>

等效于 tail -f /tmp/laser/logs/node-{nodeId}.log，实时跟踪节点日志输出。

示例
./cluster.sh log 0     # 实时查看节点 0 日志
./cluster.sh log 1     # 实时查看节点 1 日志
./cluster.sh log 2     # 实时查看节点 2 日志

按 Ctrl + C 退出日志查看。

场景：三节点集群部署

一键启动
# 编译 + 启动全部 3 个节点
./cluster.sh start-all

# 查看状态
./cluster.sh status

# 查看节点 0 日志
./cluster.sh log 0
逐个启动
# 分别启动每个节点
./cluster.sh start 0
./cluster.sh start 1
./cluster.sh start 2
关闭集群
./cluster.sh stop-all

场景：单节点开发调试

# 单节点启动，直接成为 Leader
./cluster.sh start-single

# 查看状态
./cluster.sh status

# 查看日志
./cluster.sh log 0

# 停止
./cluster.sh stop 0

场景：重启单个节点

方式一：使用 restart 命令（推荐）
# 一键重启节点 1（自动 stop + build + start）
./cluster.sh restart 1

# 确认恢复
./cluster.sh status
方式二：手动 stop + start
# 停止节点 1
./cluster.sh stop 1

# 重新编译并启动节点 1（自动编译）
./cluster.sh start 1

# 确认恢复
./cluster.sh status

JVM 参数说明

Aeron/Agrona 需要访问 JDK 内部模块，脚本自动为每个节点添加以下 --add-opens 参数：

参数用途
java.base/jdk.internal.miscAgrona UnsafeApi 访问 jdk.internal.misc.Unsafe
java.base/sun.nio.chAeron MediaDriver 访问底层 NIO channel 实现
java.base/java.nioAgrona DirectBuffer 反射访问 NIO Buffer 内部字段
java.base/java.langAgrona 反射访问 Thread 等内部实现

配置属性

以下属性定义在 application.properties 中，可通过 -D 参数在启动时覆盖：

属性默认值说明
laser.aeron.cluster.config.nodeId0当前节点编号（脚本自动覆盖）
laser.aeron.cluster.config.baseDir/tmp/laserAeron 集群数据根目录
laser.aeron.cluster.config.channelPrefixaeron:udp?endpoint=localhost:Aeron 通道前缀
laser.aeron.cluster.config.clusterNodeSize3集群节点数（start-single 覆盖为 1）
laser.aeron.cluster.config.nodeIplocalhost节点 IP 地址

常见问题

Raft 协议要求多数节点在线才能完成选举。3 节点集群至少需要 2 个节点在线。 请使用 ./cluster.sh status 确认至少 2 个节点处于 running 状态。 开发调试时可使用 ./cluster.sh start-single 单节点模式绕过此限制。

说明 JVM 未添加 --add-opens 参数。请确保通过 cluster.sh 启动节点， 脚本会自动添加所需的 4 个 --add-opens 参数。 若使用 IDE 启动，请在 VM options 中手动添加。

脚本先发送 SIGTERM，等待最多 10 秒后自动发送 SIGKILL 强制终止。 如果 PID 文件被意外删除，可手动查找进程： 
ps aux | grep laser-matching-engine
kill <PID>

/tmp 目录在系统重启后可能被清理。生产环境请修改脚本配置区的 PID_DIR 和 LOG_DIR，以及 application.properties 中的 baseDir， 指向持久化目录。
