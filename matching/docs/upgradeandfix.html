优化 1: BigDecimal → long（最高优先级）

类: MatchOrder → MatchOrderV1 | 影响: 所有价格/数量字段和运算

原因: BigDecimal 每次运算创建新对象（add/subtract/min），compareTo 有方法调用开销。long 直接用 +/-/>= 操作符，零对象分配。

V0 (修改前) BigDecimal
// MatchOrder.java
private BigDecimal delegatePrice;
private BigDecimal delegateCount;
private BigDecimal dealtCount;

public BigDecimal getRemainingQuantity() {
    return BigDecimalUtil.subtract(
        this.delegateCount, this.dealtCount);
}

public void updateFilledQuantity(BigDecimal matchedQty) {
    this.dealtCount = BigDecimalUtil.add(
        this.dealtCount, matchedQty);
    if (fullFilled()) { ... }
}

public boolean fullFilled() {
    return BigDecimalUtil.greaterOrEquals(
        this.dealtCount, this.delegateCount);
}

public BigDecimal getMatchedQty(BigDecimal oppoRemaining) {
    return this.getRemainingQuantity()
               .min(oppoRemaining);
}
V1 (修改后) long
// MatchOrderV1.java
private long delegatePrice;
private long delegateCount;
private long dealtCount;

public long getRemainingQuantity() {
    return delegateCount - dealtCount;
}

public void updateFilledQuantity(long matchedQty) {
    this.dealtCount += matchedQty;
    if (fullFilled()) { ... }
}

public boolean fullFilled() {
    return dealtCount >= delegateCount;
}

public long getMatchedQty(long oppoRemaining) {
    return Math.min(
        getRemainingQuantity(), oppoRemaining);
}

优化 2: DepthLine 保留 LinkedList（纠正）

类: DepthLine → DepthLineV1 | 变更: 仅数据类型从 BigDecimal → long

保留 LinkedList 的原因: 撮合热路径中 iterator.remove() 是 O(1)（只改指针）， 生产环境同价档位可达数万订单，若用 ArrayList，每次 remove 需移动后续所有元素 O(n)，代价极高。

V0 LinkedList + BigDecimal
// DepthLine.java
private BigDecimal price;
private List<MatchOrder> orders = new LinkedList<>();
V1 LinkedList + long
// DepthLineV1.java
private long price;
private List<MatchOrderV1> orders = new LinkedList<>();

优化 3: pendingRemoves LinkedList → ArrayList

类: MatchEngine, FokOrderProcessor

原因: 频繁 clear()+add()+iterate 模式，ArrayList 更高效。

// MatchEngine.java
private List<MatchOrder> pendingRemoves
    = new LinkedList<>();
// FokOrderProcessor.java
private List<MatchOrder> fokPendingRemoves
    = new LinkedList<>();
// MatchEngineV1.java
private List<MatchOrderV1> pendingRemoves
    = new ArrayList<>();
// FokOrderProcessorV1.java
private List<MatchOrderV1> fokPendingRemoves
    = new ArrayList<>();

优化 4: TreeMap<BigDecimal> → TreeMap<Long>

类: OrderBook → OrderBookV1

原因: TreeMap 内部排序从 BigDecimal.compareTo()（方法调用+对象访问）→ Long.compareTo()（拆箱为原生 long 比较）。

// OrderBook.java
private TreeMap<BigDecimal, DepthLine> buyOrders
    = new TreeMap<>(Collections.reverseOrder());
private TreeMap<BigDecimal, DepthLine> sellOrders
    = new TreeMap<>();
// OrderBookV1.java
private TreeMap<Long, DepthLineV1> buyOrders
    = new TreeMap<>(Collections.reverseOrder());
private TreeMap<Long, DepthLineV1> sellOrders
    = new TreeMap<>();

优化 5: 消除热路径 JSON 序列化（最大瓶颈）

类: MatchEngine → MatchEngineV1, OrderBook → OrderBookV1

原因: V0 的 printOrderBook() 在每次下单/撤单/改单后调用 JSON.toJSONString(firstEntry, true)（含 pretty-print）。即使 logback 设为 WARN，SLF4J 参数是急切求值，序列化仍然执行。这是 V0 最大的性能杀手。

V0 每次操作都序列化
// MatchEngine.java — 每个操作后都调用
orderBook.printOrderBook("placeOrder");
orderBook.printOrderBook("cancelOrder");
orderBook.printOrderBook("amendOrder");

// OrderBook.java
public void printOrderBook(String scene) {
    log.info("{} buyOne:{}, sellOne:{}",
        scene,
        JSON.toJSONString(
            buyOrders.firstEntry(), true),  // 昂贵!
        JSON.toJSONString(
            sellOrders.firstEntry(), true)); // 昂贵!
}

// MatchEngine.publishMarketHangingCmd()
log.info("[marketOrder] ...",
    JSON.toJSONString(marketOrder, true)); // 昂贵!
V1 完全移除
// MatchEngineV1.java
// printOrderBook() 调用完全移除
// 热路径零日志开销

// publishMarketHangingCmd 简化
log.debug("[marketOrder] orderId={}",
    marketOrder.getOrderId());
// debug 级别 + 无序列化

优化 6: 消除 BigDecimalUtil 调用开销

影响: 所有 V1 类

原因: V0 通过 BigDecimalUtil 包装层调用，每次都做 null 检查。V1 直接用 long 运算，零方法调用开销。

// V0: 每个运算都经过包装
BigDecimalUtil.subtract(delegateCount, dealtCount)
BigDecimalUtil.add(dealtCount, matchedQty)
BigDecimalUtil.greaterOrEquals(dealtCount, delegateCount)
BigDecimalUtil.greaterThanZero(dealtCount)
// V1: 直接原生运算
delegateCount - dealtCount
dealtCount += matchedQty
dealtCount >= delegateCount
dealtCount > 0

优化 7: deepCopy 优化

类: MatchOrder → MatchOrderV1

原因: V0 拷贝 BigDecimal 引用（不可变但仍有引用拷贝开销）。V1 拷贝的全是 long 基本类型，CPU 单条指令完成。

Bug 修复 1: MatchOrder(long orderId) 构造函数

位置: MatchOrder.java 第 122-128 行

V0 (BUG) 创建无用对象
public MatchOrder(long orderId) {
    // 创建了临时对象，设置在临时对象上!
    // this 的字段完全没有初始化!
    MatchOrder matchOrder = new MatchOrder();
    matchOrder.setOrderId(orderId);
    matchOrder.setOrderStatus(OrderStatusEnum.NEW);
    matchOrder.setDealtCount(BigDecimal.ZERO);
    matchOrder.setCancelReason(CancelReasonEnum.NONE);
}
V1 (修复) 正确初始化 this
public MatchOrderV1(long orderId) {
    this.orderId = orderId;
    this.orderStatus = OrderStatusEnum.NEW;
    this.dealtCount = 0;
    this.cancelReason = CancelReasonEnum.NONE;
}

Bug 修复 2: amendOrder setDealtCount

说明: 计划中发现的 setDealtCount(newDelegateCount) 应为 setDelegateCount(newDelegateCount)。经检查当前 V0 源码已是正确写法，V1 中同样保持正确。
