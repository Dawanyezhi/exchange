ISS-001：--release 与 --add-exports 冲突

已解决 编译阶段 2026-03-11
错误现象
[ERROR] 不允许在使用 --release 时从系统模块 java.base 导出程序包:
[ERROR] Failed to execute goal org.apache.maven.plugins:maven-compiler-plugin:3.13.0:compile
        (default-compile) on project laser-matching-engine: Compilation failure
根因分析

引入 spring-boot-starter-parent 3.4.5 后，父 POM 自动设置了： 

<maven.compiler.release>${java.version}</maven.compiler.release>

而项目的 maven-compiler-plugin 中配置了 --add-exports java.base/sun.nio.ch=ALL-UNNAMED 供 Aeron/Agrona 使用。--release 模式会锁定编译 API 到指定 JDK 版本的公开范围， 不允许通过 --add-exports 访问系统模块的内部包。

flowchart LR A["spring-boot-starter-parent"] -->|设置| B["maven.compiler.release = 17"] B -->|等价于| C["javac --release 17"] C -->|冲突| D["--add-exports java.base/sun.nio.ch"] D -->|结果| E["编译失败"] style E fill:#dc3545,color:#fff,stroke:none
解决方案

在 maven-compiler-plugin 的 <configuration> 中通过 combine.self="override" 禁用 release 参数，退回到 --source + --target 模式：

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <release combine.self="override"/>
        <source>17</source>
        <target>17</target>
        <compilerArgs>
            <arg>--add-exports</arg>
            <arg>java.base/sun.nio.ch=ALL-UNNAMED</arg>
        </compilerArgs>
    </configuration>
</plugin>
知识点
编译模式行为内部 API 访问
--release N 同时控制 source、target 和 bootclasspath，严格限定到 JDK N 的公开 API 禁止
--source N + --target N 仅控制语法级别和字节码版本，使用当前 JDK 的完整 classpath 允许（配合 --add-exports）

根因分析

在解决 ISS-001 时，最初尝试在 <properties> 中将 maven.compiler.release 设置为空字符串：

ISS-005：从 Parent 继承改为 BOM 导入

已解决 架构调整 2026-03-11
背景

原先通过 <parent> spring-boot-starter-parent 继承 Spring Boot 配置。 这种方式占用了唯一的 parent 位置，不便于将来引入公司内部父 POM 或多模块结构。 改为 BOM（dependencyManagement + import）方式更灵活。

两种方式对比
特性Parent 继承BOM 导入
依赖版本管理 自动 自动（通过 BOM）
插件版本 & 默认配置 自动继承 需手动指定（如 spring-boot-maven-plugin 的 version 和 repackage goal）
parent 位置 被占用 空闲，可用于公司内部 POM
maven.compiler.release parent 自动设置，需 override 无影响，可直接使用 source/target
改动要点
改动前（Parent 继承）
<parent>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-parent</artifactId>
    <version>3.4.5</version>
</parent>
改动后（BOM 导入）
<properties>
    <spring-boot.version>3.4.5</spring-boot.version>
</properties>

<dependencyManagement>
    <dependencies>
        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>${spring-boot.version}</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>
额外需要补充的配置
<!-- BOM 模式下需显式指定 spring-boot-maven-plugin 版本和 repackage goal -->
<plugin>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-maven-plugin</artifactId>
    <version>${spring-boot.version}</version>
    <executions>
        <execution>
            <goals>
                <goal>repackage</goal>
            </goals>
        </execution>
    </executions>
</plugin>
附带收益：改为 BOM 后，ISS-001 中 <release combine.self="override"/> 的 hack 不再需要了， 因为不再有 parent 自动设置 maven.compiler.release。编译插件配置更简洁。

ISS-006：@Value 与 @ConfigurationProperties 混用导致 TypeMismatchException

已解决 运行时 2026-03-15
错误现象
Caused by: org.springframework.beans.TypeMismatchException:
    Failed to convert value of type 'java.lang.String' to required type
    'java.lang.Integer'; For input string: "nodeId"
    at org.springframework.beans.TypeConverterSupport.convertIfNecessary
    at org.springframework.beans.factory.support.DefaultListableBeanFactory
        .doResolveDependency
根因分析

AeronClusterConfiguration 类同时使用了 @ConfigurationProperties 和 @Value 注解， 且 @Value 的写法错误： 

@Configuration
@ConfigurationProperties(prefix = "laser.aeron.cluster.config")
public class AeronClusterConfiguration {

    @Value("nodeId")         // ← 错误！传入字面量字符串 "nodeId"
    private Integer nodeId;

    @Value("baseDir")        // ← 错误！传入字面量字符串 "baseDir"
    private String baseDirectory;
}

问题出在三处：

#问题说明
1 @Value("nodeId") 缺少 ${} 占位符 Spring 将 "nodeId" 作为字面量注入，而非读取配置属性。字面量 "nodeId" 无法转为 Integer，抛出 TypeMismatchException
2 @Value 与 @ConfigurationProperties 混用 两者是不同的属性绑定机制，同时使用会导致绑定冲突和语义混乱
3 字段名 baseDirectory 与配置属性名 baseDir 不匹配 @ConfigurationProperties 通过字段名与配置属性名进行自动绑定，名称不一致无法自动匹配
flowchart TB A["@Value("nodeId")"] -->|注入字面量| B["String "nodeId""] B -->|类型转换| C["Integer.parseInt("nodeId")"] C -->|失败| D["TypeMismatchException"] E["@Value("${laser.aeron...nodeId}")"] -->|读取配置| F["String "0""] F -->|类型转换| G["Integer 0 ✓"] style A fill:#dc3545,color:#fff,stroke:none style D fill:#dc3545,color:#fff,stroke:none style E fill:#198754,color:#fff,stroke:none style G fill:#198754,color:#fff,stroke:none
@Value 与 @ConfigurationProperties 对比
特性@Value@ConfigurationProperties
绑定方式 逐个字段指定完整属性路径 ${key} 通过 prefix + 字段名自动映射
松散绑定 不支持 支持（kebab-case、camelCase 等）
类型安全 弱（字符串注入后转换） 强（编译期可配合 @Validated）
适用场景 单个值注入 结构化配置组（推荐）
解决方案
  1. 移除所有 @Value 注解，由 @ConfigurationProperties 统一绑定
  2. 将字段名 baseDirectory 改为 baseDir，与配置属性名保持一致
  3. 添加 Lombok @Getter / @Setter（@ConfigurationProperties 需要通过 setter 注入属性值）
修改前
@Configuration
@ConfigurationProperties(prefix = "laser.aeron.cluster.config")
public class AeronClusterConfiguration {

    @Value("nodeId")
    private Integer nodeId;

    @Value("baseDir")
    private String baseDirectory;
}
修改后
@Configuration
@ConfigurationProperties(prefix = "laser.aeron.cluster.config")
@Getter
@Setter
public class AeronClusterConfiguration {

    private Integer nodeId;

    private String baseDir;
}
最佳实践：对于结构化配置，优先使用 @ConfigurationProperties 而非 @Value。 前者支持松散绑定、类型安全校验、嵌套对象映射，是 Spring Boot 推荐的配置绑定方式。 @Value 更适合注入单个简单值（如 @Value("${server.port}")）。

ISS-007：ClusterNodeBoot 集群启动器多处隐患

已解决 运行时 2026-03-16
问题概述

Code Review 发现 ClusterNodeBoot 中存在 5 处 Bug，涉及端点格式拼接错误、 目录路径缺失、Spring 启动阻塞、资源泄漏及目录创建失败等问题。

Bug 清单
#问题严重程度位置
1 buildClusterMembers() 拼接端点缺少 : 分隔符，生成 localhost9000 而非 localhost:9000 致命 ClusterNodeBoot:168-180
2 clusterDir 路径仅使用前缀 "node-"，缺少 nodeId 拼接，所有节点共享同一目录 致命 ClusterNodeBoot:40
3 @PostConstruct 中 Thread.currentThread().join() 永久阻塞 Spring Bean 初始化，导致 ApplicationContext 无法启动完成 致命 ClusterNodeBoot:32,127
4 ShutdownHook 仅打印日志，无实际关闭 Aeron 组件的逻辑，try-with-resources 的 close() 因 join() 永远不会执行 中等 ClusterNodeBoot:123-125
5 mkdir() 仅创建最后一级目录，父目录不存在时静默失败 低 ClusterNodeBoot:48
Bug-1：端点缺少冒号分隔符

Aeron Cluster 的 clusterMembers 要求端点格式为 hostname:port， 但 buildClusterMembers() 中直接拼接 nodeIp + port，缺少 :。

flowchart LR A["nodeIp = "localhost""] -->|直接拼接 port| B["localhost9000"] B -->|Aeron 解析| C["端点格式错误"] A -->|拼接 ":" + port| D["localhost:9000"] D -->|Aeron 解析| E["正确 ✓"] style B fill:#dc3545,color:#fff,stroke:none style C fill:#dc3545,color:#fff,stroke:none style D fill:#198754,color:#fff,stroke:none style E fill:#198754,color:#fff,stroke:none 
// 改前
memberUri.append(getNodeIp()).append(getCLIENT_FACING_PORT_BASE() + i);
// → "localhost9000"

// 改后
memberUri.append(getNodeIp()).append(":").append(getCLIENT_FACING_PORT_BASE() + i);
// → "localhost:9000"
Bug-2：clusterDir 缺少 nodeId
// 改前: 所有节点共享 /tmp/laser/node-
new File(baseDir, CLUSTER_DIR_PREFIX)

// 改后: 每个节点独立 /tmp/laser/node-0, /tmp/laser/node-1, ...
new File(baseDir, CLUSTER_DIR_PREFIX + nodeId)
Bug-3：@PostConstruct 阻塞 Spring 启动

@PostConstruct 在 Spring Bean 初始化阶段执行。方法内调用 Thread.currentThread().join() 使当前线程等待自己结束（永远阻塞）， 导致 Spring ApplicationContext 初始化无法完成，Embedded Tomcat 不会启动。

flowchart TB subgraph "改前（阻塞）" A1["Spring init"] --> B1["@PostConstruct"] B1 --> C1["launch Aeron"] C1 --> D1["Thread.join()"] D1 -->|永远阻塞| E1["Spring 启动不完成 ✗"] end subgraph "改后（非阻塞）" A2["Spring init"] --> B2["@PostConstruct"] B2 --> C2["启动 daemon 线程"] C2 -->|立即返回| D2["Spring 启动完成 ✓"] C2 -->|daemon 线程| E2["launch Aeron"] end style E1 fill:#dc3545,color:#fff,stroke:none style D2 fill:#198754,color:#fff,stroke:none 
// 改后: 使用 daemon 线程启动集群，不阻塞 Spring 初始化
@PostConstruct
void initCluster() {
    Thread clusterThread = new Thread(this::launchCluster, "aeron-cluster-main");
    clusterThread.setDaemon(true);
    clusterThread.start();
}
Bug-4：ShutdownHook 无关闭逻辑 → 改用 @PreDestroy

原 ShutdownHook 仅打印日志，Aeron 组件（MediaDriver、Archive、ConsensusModule、ServiceContainer） 在 JVM 退出时不会被正确关闭。改用 Spring 的 @PreDestroy 生命周期回调， 配合 Agrona 的 CloseHelper.quietClose() 按序释放资源。 

// 改前: ShutdownHook 仅打日志
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    log.info("shutdown handling....");
}));

// 改后: Spring 容器关闭时自动调用
@PreDestroy
void shutdown() {
    log.info("Shutting down Aeron cluster...");
    CloseHelper.quietClose(clusteredServiceContainer);
    CloseHelper.quietClose(clusteredMediaDriver);
    log.info("Aeron cluster shutdown complete.");
}
Bug-5：mkdir() → mkdirs()
// 改前: 父目录 /tmp/laser 不存在时创建失败
clusterDir.mkdir()

// 改后: 递归创建所有缺失的父目录
clusterDir.mkdirs()
教训：Aeron 集群组件的生命周期应独立于 Spring Bean 初始化流程。 使用独立 daemon 线程启动集群，通过 @PreDestroy 优雅关闭， 既不阻塞 Spring 启动，又能在容器关闭时正确释放底层资源。

ISS-008：cluster.sh 空数组展开导致 start 命令失效

已解决 运行时 2026-03-16
错误现象

./cluster.sh start-single 正常工作，但 ./cluster.sh start 0 和 ./cluster.sh start-all 均无法启动节点，脚本静默退出。

根因分析

do_start_node() 函数接收可选的额外 JVM 参数，内部通过 shift 后将剩余参数赋值给数组： 

do_start_node() {
    local nid="$1"; shift
    local extra_props=("$@")   # start 0 时为空数组
    ...
    nohup java ... "${extra_props[@]}" -jar ...
    #             ^^^^^^^^^^^^^^^^^^^ 空数组展开触发错误
}

脚本头部设置了 set -euo pipefail，其中 -u（nounset） 要求所有变量必须已定义。macOS 自带的 bash 3.2 将空数组 "${arr[@]}" 视为未绑定变量，触发 unbound variable 错误并退出。 bash 4.4+ 修复了此行为，空数组展开不再报错。

调用方式extra_propsbash 3.2 + set -u结果
start-single ("-Dlaser...=1") 非空 正常展开 正常
start 0 () 空数组 unbound variable 脚本退出
flowchart LR A["start 0"] --> B["do_start_node 0"] B --> C["shift → $@ 为空"] C --> D["extra_props=()"] D --> E[""${extra_props[@]}""] E -->|bash 3.2 + set -u| F["unbound variable → 脚本退出"] style F fill:#dc3545,color:#fff,stroke:none
解决方案

去掉中间变量 extra_props，直接使用 ${@+"$@"} 展开剩余参数。 这是 bash 的标准兼容写法：当 $@ 为空时不展开，非空时正常展开，兼容所有 bash 版本。

# 改前: 空数组在 bash 3.2 + set -u 下报错
local extra_props=("$@")
nohup java ... "${extra_props[@]}" -jar ...

# 改后: ${@+"$@"} 兼容所有 bash 版本
nohup java ... ${@+"$@"} -jar ...
同步进行的 Linux 兼容性加固
#修改原因
1 #!/bin/bash → #!/usr/bin/env bash 部分 Linux 发行版（NixOS、Alpine、容器镜像）bash 不在 /bin/bash，env 方式从 PATH 查找更通用
2 echo -e → printf echo -e 行为依赖 bash 的 xpg_echo 选项，printf 是 POSIX 标准，跨平台行为一致
Shell 脚本跨平台经验： (1) 使用 #!/usr/bin/env bash 替代硬编码路径； (2) 用 printf 替代 echo -e； (3) set -u 下避免直接展开可能为空的数组，用 ${arr[@]+"${arr[@]}"} 或 ${@+"$@"} 防御。

ISS-009：onStart() 中 cluster.memberId() 返回 -1

已解决 运行时 2026-03-16
错误现象
onStart memberId:-1, role:FOLLOWER

启动集群节点时，onStart() 方法中调用 cluster.memberId() 始终返回 -1， 即使在 application.properties 和 -D 参数中已经正确设置了 nodeId。

根因分析

这是 Aeron Cluster 的正常行为，而非配置错误。 ClusteredService.onStart() 回调发生在 ClusteredServiceContainer 初始化阶段， 此时 Service Container 尚未完成与 Consensus Module 的连接握手。

cluster.memberId() 的值由 Consensus Module 在握手完成后写入， 在 onStart() 阶段该字段尚未被赋值，因此返回默认值 -1。

sequenceDiagram participant SC as ServiceContainer participant CM as ConsensusModule participant CS as ClusteredService SC->>CS: onStart(cluster, snapshot) Note over CS: cluster.memberId() = -1
握手尚未完成 SC->>CM: 连接握手 CM-->>SC: 返回 memberId Note over CS: 此后 memberId() 可用 CM->>CS: onRoleChange(newRole) Note over CS: cluster.memberId() = 正确值
Aeron 生命周期回调中 memberId 可用性
回调方法memberId()说明
onStart() -1（不可用） Service Container 尚未与 Consensus Module 完成握手
onRoleChange() 正确值 已完成握手，memberId 已被赋值
onSessionMessage() 正确值 正常运行阶段
onTerminate() 正确值 关闭阶段，仍可用
解决方案

在 onStart() 中使用 Spring 配置中的 nodeId 替代 cluster.memberId()； 在 onRoleChange() 等后续回调中正常使用 cluster.memberId()。 

@Resource
private AeronClusterConfiguration aeronClusterConfiguration;

@Override
public void onStart(Cluster cluster, Image snapshotImage) {
    this.cluster = cluster;
    this.currentRole = cluster.role();
    // onStart 阶段 cluster.memberId() 返回 -1，使用配置中的 nodeId
    log.info("onStart nodeId:{}, role:{}",
        aeronClusterConfiguration.getNodeId(), currentRole);
}

@Override
public void onRoleChange(Cluster.Role newRole) {
    Cluster.Role previousRole = this.currentRole;
    this.currentRole = newRole;
    // onRoleChange 阶段 memberId() 已可用
    log.info("onRoleChange memberId:{}, {} -> {}",
        cluster.memberId(), previousRole, newRole);
}
关键认知：Aeron Cluster 的 onStart() 回调时机早于 Consensus Module 握手完成， 因此 cluster.memberId() 在此阶段不可用。如需在启动时获取节点标识， 应通过外部配置（如 Spring @ConfigurationProperties）传入。

ISS-010：ClusterEvent 心跳超时被误报为 ERROR

已解决 运行时 2026-03-16
错误现象
[consensus-module-0-1] ERROR c.l.matching.cluster.ClusterNodeBoot - ConsensusModule occurred errorHandler.
io.aeron.cluster.client.ClusterEvent: WARN - leader heartbeat timeout

手动关掉 Leader 节点后，某个 Follower 节点报 ERROR 级别日志。 实际上这是 Raft 协议检测到 Leader 失联后触发选举的正常行为。

根因分析

Aeron Cluster 将集群事件（如心跳超时、选举等）通过 ErrorHandler 回调传递， 而原来的 error handler 对所有回调统一使用 log.error 输出。

关键在于 ClusterEvent 的继承链： 

ClusterEvent → AeronEvent → AeronException → ... → Throwable

AeronEvent 默认 category 为 WARN，且不携带 stack trace（轻量级事件）， 属于 Raft 协议正常运行的一部分，不应以 ERROR 级别记录。

flowchart LR A["Leader 被关闭"] --> B["Follower 心跳超时"] B --> C["ClusterEvent
WARN 级别"] C -->|旧代码| D["log.error 输出
误报为严重错误"] C -->|新代码| E["log.warn 输出
正确反映事件级别"] style D fill:#dc3545,color:#fff,stroke:none style E fill:#198754,color:#fff,stroke:none
解决方案

将 4 个独立的 ErrorHandler lambda 合并为统一的 buildErrorHandler(String component) 方法， 根据异常类型智能路由日志级别：

Throwable 类型日志级别说明
ClusterEvent WARN 集群事件（心跳超时、选举等），Raft 正常行为
其他 Throwable ERROR 真正的组件异常 
private ErrorHandler buildErrorHandler(String component) {
    return throwable -> {
        if (throwable instanceof ClusterEvent) {
            log.warn("[{}] {}", component, throwable.getMessage());
        } else {
            log.error("[{}] errorHandler.", component, throwable);
        }
    };
}
经验：Aeron 的 ErrorHandler 不仅传递真正的异常，也传递集群事件。 通过 instanceof ClusterEvent 区分后，可避免误报并保留有效的告警信息。

实现方式

MatchEngineClusteredService 中的关键设计：

@Service
public class MatchEngineClusteredService implements ClusteredService {

    @Resource
    private AeronClusterConfiguration aeronClusterConfiguration;

    // Cluster 引用，onStart 时保存，后续可查询 memberId/logPosition 等
    private volatile Cluster cluster;

    // 当前角色，volatile 保证多线程可见性
    @Getter
    private volatile Cluster.Role currentRole = Cluster.Role.FOLLOWER;

    @Override
    public void onStart(Cluster cluster, Image snapshotImage) {
        this.cluster = cluster;
        this.currentRole = cluster.role();
        // 注意：onStart 阶段 cluster.memberId() 返回 -1（见 ISS-009）
        log.info("onStart nodeId:{}, role:{}",
            aeronClusterConfiguration.getNodeId(), currentRole);
    }

    @Override
    public void onRoleChange(Cluster.Role newRole) {
        Cluster.Role previousRole = this.currentRole;
        this.currentRole = newRole;
        log.info("onRoleChange memberId:{}, {} -> {}",
            cluster.memberId(), previousRole, newRole);
    }

    public boolean isLeader() {
        return currentRole == Cluster.Role.LEADER;
    }
}

依赖管理原则

引入 spring-boot-starter-parent 后，以下原则可以避免大部分版本冲突问题：

#原则说明
1 Spring Boot 已托管的依赖不要手动指定版本 Logback、SLF4J、Jackson、Tomcat、JUnit 等由 parent POM 统一管理，手动指定版本会覆盖托管版本，容易产生不兼容
2 非 Spring Boot 托管的依赖必须显式指定版本 Aeron、Agrona、gRPC、Protobuf、Vert.x 等不在 Spring Boot BOM 中，必须在 <properties> 中统一管理版本号
3 使用 mvn dependency:tree 排查冲突 出现 NoClassDefFoundError 或 ClassNotFoundException 时，先用依赖树定位实际加载的版本
4 覆盖父 POM 配置用 combine.self="override" 不要用空字符串覆盖，应使用 Maven 标准的 POM 合并指令
Spring Boot 3.4.5 托管的常见依赖（部分）
logback-classic     → 1.5.x     ← 不要手动指定
slf4j-api           → 2.0.x     ← 不要手动指定
jackson-databind    → 2.18.x    ← 不要手动指定
junit-jupiter       → 5.11.x    ← 不要手动指定
mockito-core        → 5.14.x    ← 不要手动指定
tomcat-embed-core   → 10.1.x    ← 不要手动指定

aeron-all           → (无托管)  ← 必须手动指定
agrona              → (无托管)  ← 必须手动指定
grpc-netty-shaded   → (无托管)  ← 必须手动指定
protobuf-java       → (无托管)  ← 必须手动指定
