1. 概览

本手册介绍 Sprint 4 新增的上下币与开关交易控制面能力。

能力SBE Command触发方式
上币 (LIST)UpDownSymbolCommand op=1HTTP / cluster.sh list-symbol
下币 (DELIST)UpDownSymbolCommand op=2HTTP / cluster.sh delist-symbol
开启交易TradeSwitchCommand switchOn=trueHTTP / cluster.sh enable-trade
关闭交易TradeSwitchCommand switchOn=falseHTTP / cluster.sh disable-trade
颗粒度对齐：上下币 / 开关交易是控制面命令，与下单 / 撮合 / 撤单（数据面）严格区分：
  • 不参与 serialNum 连续性校验（生产部署可允许运维操作和业务命令异步并发）
  • 仍使用 cluster 共识时间戳，保证 3 节点状态机确定性
  • 仍生成 MatchResult 事件 (UpDownSymbolResult / TradeSwitchResult) 投递到 MDC

2. 核心概念

2.1 symbolCode 与 symbolName

一一对应：

  • symbolCode (uint16) = 全局唯一整数 ID。例: 1
  • symbolName (varString) = 唯一名字。例: "btc-usdt"
  • 两者绑定，注册后不可修改

2.2 SymbolConfig vs MatchConfig

配置位置含义
SymbolConfigMatchContext.symbolConfigMap币对元数据 (code/name/baseCoin/quoteCoin)。下币时移除
MatchConfigMatchEngineState.configMap交易开关 (enabled)。上币默认 false, 需要 enable-trade 才能下单
OrderBookMatchContext.orderBookMap订单簿，上币时预创建空实例，下币时移除

2.3 上下币与开关交易的关系

stateDiagram-v2 [*] --> 未注册: 初始 未注册 --> 已上币_未启用: list-symbol (默认 enabled=false) 已上币_未启用 --> 已上币_已启用: enable-trade 已上币_已启用 --> 已上币_未启用: disable-trade 已上币_未启用 --> 未注册: delist-symbol 已上币_已启用 --> 未注册: delist-symbol (内部先 disable 再删除)

公共头 + op + symbolCode + baseCoinId + quoteCoinId + symbolName

公共头 + symbolCode + switchOn + symbolId

6. cluster.sh 命令

./cluster.sh list-symbol <code> <name> <baseCoin> <quoteCoin>
./cluster.sh delist-symbol <code>
./cluster.sh enable-trade <code>
./cluster.sh disable-trade <code>
前置条件：cluster.sh 上述命令通过 mock-counter (10880) 转发到集群， 执行前请确认 mock-counter 已运行 (./client.sh start)。

实战示例 — dogeUsdt 全周期

# 1. 启动集群和 mock-counter
./cluster.sh start-all
./client.sh start

# 2. 上币
./cluster.sh list-symbol 3 doge-usdt 100 200

# 3. 开启交易
./cluster.sh enable-trade 3

# 4. 业务下单
./client.sh place 3 0.5 100 BUY

# 5. 关闭交易 (将拒绝后续下单)
./cluster.sh disable-trade 3

# 6. 下币 (彻底移除)
./cluster.sh delist-symbol 3

8. 停交易事件 (Sprint 4 关键反馈)

当用户对未启用交易的币对发起 place / amend / cancel 时，必须主动推送一个 TradeSwitchResult(switchOn=false) 事件给上游，避免 silent reject。

实现位置

CommandDispatcher.emitTradeStoppedEvent

private void emitTradeStoppedEvent(long timestamp, int symbolCode, String symbolName, long requestSerialNum) {
    eventsHelper.beginRequest(requestSerialNum, timestamp);
    eventsHelper.appendTradeSwitch(symbolCode, symbolName, false);  // switchOn=false
    flushAndPersist();
}

触发时机

  • handlePlace / handleAmend / handleCancel
  • serialNum 校验通过 + symbolId 存在
  • 但 MatchConfig 不存在或 enabled=false

9. 确定性约束

维度保证
3 节点状态机一致所有节点都在状态机线程内执行 SymbolService.listSymbol/delistSymbol/switchTrade
事件时间戳使用 cluster.time() 共识时间戳, 不用 System.currentTimeMillis
MDC 广播仅 LEADER 推送, FOLLOWER 仅持久化, 避免 3 副本重复广播
幂等重复 LIST 不会重置已开启的 enabled; DELIST 不存在的 symbol 不抛异常仍发事件
raftlog 回放kill-restart 后 raftlog 自动 replay 上下币命令, 状态完全恢复

10. FAQ

Q: 上币后为什么不能立即交易？

A: 用户要求"需要生成对应的matchConfig，默认enable为false"。这是双重保护——上币是元数据注册，开启交易是业务上线，需要分别确认。这样同一个币对在不同环境（test/staging/prod）可以独立控制启停。

Q: 下币时已挂的订单怎么处理？

A: 当前实现直接移除 OrderBook，已挂订单会丢失。生产环境建议在下币前先调用 disable-trade，等所有用户撤单或人工撤单后再 delist。Sprint 5 候选：增加 delist 前自动撤单的开关。

Q: 上下币命令为什么不参与 serialNum 校验？

A: 因为它们是控制面命令——由运维端发起、低频、与业务流序列无关；而 serialNum 校验是为了检测数据面下单流的丢包。两条流在 SBE 层面通过 templateId 区分，校验路径独立，避免运维操作影响业务串行性。

Q: 关闭交易后挂在簿上的订单怎么样？

A: 仍然在 OrderBook 中，但用户既不能新下单也不能改/撤单 (会收到 TradeSwitchResult(off) 事件)。 一旦再 enable-trade 就恢复正常。

Q: 3 节点中只有 LEADER 才执行上币吗？

A: 否。所有节点都执行 SymbolService.listSymbol / delistSymbol / switchTrade —— 这是状态机复制的本质要求。但只有 LEADER 把 MatchResult 推到 MDC（避免 3 副本重复广播）。
