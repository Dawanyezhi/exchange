# 编码计划

Laser 撮合引擎 — Request 连续性 / MatchResult 体系 / 持久化 / 快照

Context

当前 laser-matching-engine 存在四块能力缺失：
1. Request 无序号与连续性校验：PlaceOrder/Amend/CancelOrderRequest 之间无继承、无
   serialNum，服务端无法检测 raftlog 回放或客户端丢包。
2. 无统一 MatchResult 体系：撮合过程只有 MatchOrder 内部状态变化，下游清算/对账无法消费。需要参考
   exchange-core 的 EventsHelper 模式，以 SBE 直接产出 5 种事件。
3. 无结果持久化：撮合结果只存在内存，断电或重启全部丢失。
4. 无快照能力：OrderBook / MatchContext / 已处理 serialNum 无法持久化恢复，Aeron raftlog
   长度无法截断。

目标：两个 sprint 闭环交付上述 4 块，保证状态机确定性（回放同 request 必产生同 result 序列，时间戳用
aeron cluster 共识时钟）。

1.3 Part3 — resultRepoModule 持久化

包位置：com.laser.exchange.matching.resultRepoModule

设计（内聚单包）：
- ResultArchiveWriter — 包装 AeronArchive 的 publish，专用 streamId（比如 100），独立 channel 避免和
  raftlog 串流
- ResultRepository (interface) — 抽象出 persist(List<MatchResult>) / readAllFrom(long startPos) /
  getMaxResultSerialNum()
- ArchiveResultRepository (impl) — 调 ResultArchiveWriter，保证写入顺序 = resultSerialNum 顺序
- ResultRepoConfig (@Configuration) — 注入 archive 实例

确定性要求：

## 快照恢复逻辑：
  
  快照时刻的 OrderBook 一定是稳定无交叉状态（撮合状态机单线程完成一个 request                           
  后才有机会快照）。所以恢复是"结构拷贝 + 状态回填"，不应触发任何业务逻辑：
  
  OrderBookStart → new OrderBook → addOrderBook                                                           
  MatchOrderEntry → 解出含 dealtCount 的 MatchOrder → orderBook.addOrder
  
  底层逻辑：snapshot 是 state serialization，不是 event replay。event replay 是 raftlog 的职责——Aeron   
  Cluster 的恢复模型正是 snapshot + raftlog catchup：
  
  load snapshot (state dump) → 把 OrderBook 原样回填                                                    
  ↓                                                                                                 
  replay raftlog from snapshot position → 重新 dispatch 快照点之后的 request（此时才会走 engine.place）
  
  如果 snapshot 恢复阶段也走 engine.place，等于同一段 request 被撮合两次——状态机确定性直接破坏。
  
  结论：当前 SnapshotManager 用 orderBook.addOrder 是对的，保持不变。
