总览

本次迭代为撮合引擎补齐四块基础能力：Request 连续性校验 · MatchResult 事件体系 · 结果持久化 · 状态快照与恢复。所有改动均建立在 Aeron Cluster 状态机之上，严格保证回放确定性。

核心约束

  • 所有时间戳来自 Aeron Cluster 共识时钟，严禁 System.currentTimeMillis()
  • 状态机线程内禁用 AtomicLong / synchronized；普通 long 直接 +1
  • 事件序列固定顺序：PLACE → MATCH×N → CANCEL?
  • resultSerialNum 跨 symbol 全局单调递增，无跳号

Part 1 · Request 连续性校验

客户端按 serialNum=1,2,3,... 发送，服务端入口校验 newSerialNum == lastSerialNum + 1。不连续即丢包，生成 SystemErrorCode=SERIAL_NUM_NOT_CONTINUOUS 的错误结果。

Part 3 · resultRepoModule 持久化

抽象 ResultRepository + 两套实现：InMemoryResultRepository（测试）与 ArchiveResultRepository（生产）。生产实现复用 Aeron Cluster 内置 Archive，独立 streamId (默认 1001) 隔离 raftlog/snapshot 流。

streamId=1001]

背压策略

Archive offer 返回 BACK_PRESSURED / NOT_CONNECTED 时 busy-spin (Thread.onSpinWait + 偶发 parkNanos(1))，状态机线程内严禁 sleep / wait。

关键决策：恢复时为什么不调用 engine.placeOrder？

因为快照是 state serialization，不是 event replay。直接 orderBook.addOrder(...) 才能：

  • 保留每个订单的 dealtCount（部分成交历史）
  • 避免恢复中途产生的瞬时交叉触发重复撮合
  • 避免 POST_ONLY 在恢复中被误撤、市价单被误入队
  • 避免 EventsHelper 重复生成 PLACE 事件膨胀 resultSerialNum

Aeron Cluster 模型 = snapshot 恢复状态 + raftlog catchup 重放后续 request，二者职责严格分离。

Part 6 · MatchResult MDC 广播 (Sprint 3)

MDC = Multi-Destination-Control，Aeron 的动态广播通道。撮合 LEADER 把每条 MatchResult 通过 UDP MDC publication 推给订阅者（mock-counter / 清算 / 风控）。

拓扑与端口规则

节点MDC control 端口客户端 subscriber 端口
node-040456 (base + 0)40500
node-140457 (base + 1)40501
node-240458 (base + 2)40502

数据流

flowchart LR subgraph Leader["LEADER 节点"] Dispatcher[CommandDispatcher] Role[ClusterRoleHolder] Repo[ArchiveResultRepository
每节点都写] Broadcaster[ResultMdcBroadcaster
仅 LEADER] end subgraph Follower["FOLLOWER 节点"] Repo2[ArchiveResultRepository
每节点都写] BCOff[Broadcaster
skip by isLeader check] end Dispatcher --> Repo Dispatcher -->|isLeader?| Role Dispatcher -.-> Broadcaster Broadcaster -->|UDP MDC aeron:udp?control=...| MockCounter[mock-counter
ResultMdcSubscriber] MockCounter --> Log[SBE 解码 → MDC-RESULT 日志]

关键不变量

  • 所有节点都执行 resultRepository.persist()（状态机确定性）
  • 仅 LEADER 执行 broadcaster.broadcastBatch()（避免 3 副本重复）
  • FOLLOWER 的 broadcaster publication 也会绑定端口（冷备），role 切换后立即可用

升级思路 (Sprint 4 候选)

当前 MDC 是 live broadcast（撮合完成后直接推）。未来可单独抽出进程，从 archive 异步 replay 到 MDC，支持历史回放 / 断线补发场景。

Part 7 · 依赖解耦 · ClusterRoleHolder (Sprint 3)

问题：循环依赖

初版实现中 CommandDispatcher 为判断 LEADER 直接注入整个 MatchEngineClusteredService，Spring 启动报错：

The dependencies of some of the beans ... form a cycle:
  clusterNodeBoot
┌─────┐
|  matchEngineClusteredService
↑     ↓
|  commandDispatcher
└─────┘

重构：单向数据流

flowchart LR CS[ClusteredService
onStart/onRoleChange] -->|write| RH[ClusterRoleHolder
volatile Role] CS -->|调用入口| CD[CommandDispatcher] CD -->|read isLeader| RH SS[SnapshotScheduler] -->|read| RH SC[SnapshotController] -->|read| RH

底层逻辑

CommandDispatcher 只需知道 当前是不是 LEADER，而非整个 ClusteredService。抽出 ClusterRoleHolder 这个轻量组件后：

  • 写入端（ClusteredService）单一；读取端（Dispatcher / Scheduler / Controller）多个；职责解耦
  • 依赖箭头单向，编码层面完全消除循环
  • 符合接口隔离原则（ISP），颗粒度合适

确定性约束总览

类禁用替代
SerialNumValidatorAtomicLong普通 long + 1
MatchResultEventsHelperAtomicLong / System.currentTimeMillis普通 long; cluster.time()
ArchiveResultRepositorypark/sleep/waitThread.onSpinWait()
SnapshotManagerHashMap 遍历(顺序不确定)按 SBE 消息顺序写出
