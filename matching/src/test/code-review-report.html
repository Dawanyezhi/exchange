项目概述

Laser Matching Engine 是一个高性能的金融交易撮合引擎，基于 Java 17 构建， 采用 Aeron Cluster 实现分布式共识，支持多种订单类型和自成交保护机制。 代码质量优良，测试覆盖全面，适合作为生产级撮合系统的基础。

订单类型对比

类型价格成交策略未成交处理实现状态
GTC 指定价格 价格交叉时撮合 挂单等待 完整
IOC 指定价格 立即撮合可成交部分 撤销剩余 完整

POST_ONLY 指定价格 拒绝立即成交 挂单或撤销 完整
MARKET 市场最优价 立即撮合 挂单等待 部分
FOK 两阶段设计亮点:
  1. Phase 1 (预撮合): 使用订单深拷贝模拟成交，不修改订单簿
  2. Phase 2 (真实撮合): 确认可完全成交后，执行实际撮合
这种设计确保了 FOK 的原子性语义，避免部分成交后回滚的复杂性。

撮合优先级规则

价格优先
  • 买单: 价格从高到低 (TreeMap 降序)
  • 卖单: 价格从低到高 (TreeMap 升序)
// 买盘: 降序排列
buyOrders = new TreeMap<>(Comparator.reverseOrder());
// 卖盘: 默认升序
sellOrders = new TreeMap<>();
时间优先
  • 同价格档位内，先到先得 (FIFO)
  • 使用 LinkedList 保证插入顺序
// DepthLine 使用 LinkedList
private List<MatchOrder> orders = new LinkedList<>();

价格交叉判断

// OrderBook.isCross() 核心逻辑
public boolean isCross(MatchOrder newOrder) {
    if (newOrder.isBuy()) {
        BigDecimal bestAsk = getBestAskPrice();
        // 买单价格 >= 卖盘最低价 = 交叉
        return bestAsk != null && newOrder.getDelegatePrice().compareTo(bestAsk) >= 0;
    } else {
        BigDecimal bestBid = getBestBidPrice();
        // 卖单价格 <= 买盘最高价 = 交叉
        return bestBid != null && newOrder.getDelegatePrice().compareTo(bestBid) <= 0;
    }
}

STP 策略详解

策略触发条件处理方式适用场景
DEFAULT - 不进行任何保护 允许自成交的场景
CANCEL_TAKER 同账户订单交叉 撤销主动方 (新订单) 保护已挂单，优先做市
CANCEL_MAKER 同账户订单交叉 撤销被动方 (挂单) 保护新订单成交
CANCEL_BOTH 同账户订单交叉 双方都撤销 严格防止自成交
FOK 特殊处理: FOK 订单在 STP 触发时不会立即撤销，而是返回控制信号 (OP_BREAK/OP_CONTINUE)， 由 FokOrderProcessor 在 Phase 1 预撮合阶段统一处理，确保 FOK 语义的原子性。

架构设计 - 设计模式

单例模式 (Enum Singleton)

MatchCoreService 使用枚举实现线程安全单例

public enum MatchCoreService {
    INSTANCE;

    public void doMatch(...) {
        // 核心撮合逻辑
    }
}
优点: 线程安全、防反射攻击、简洁
策略模式 (Strategy Pattern)

STP 策略通过枚举实现多种处理方式

switch (taker.getStpStrategyEnum()) {
    case CANCEL_TAKER: ...
    case CANCEL_MAKER: ...
    case CANCEL_BOTH: ...
}
优点: 策略独立、易扩展
建造者模式 (Builder Pattern)

MatchOrder 使用 Lombok @Builder

MatchOrder order = MatchOrder.builder()
    .orderId(1001L)
    .delegatePrice(new BigDecimal("100"))
    .delegateCount(new BigDecimal("10"))
    .build();
优点: 参数清晰、构建灵活
迭代器模式 (Iterator Pattern)

安全遍历并删除订单簿元素

Iterator<MatchOrder> it = depthLine.iterator();
while (it.hasNext()) {
    MatchOrder order = it.next();
    if (order.isMatchOver()) {
        it.remove(); // 安全删除
    }
}
优点: 避免 ConcurrentModificationException

架构设计 - 数据结构选择

数据结构使用场景时间复杂度选择原因
TreeMap<BigDecimal, DepthLine> 买/卖订单簿 查找 O(log n) 自动排序、支持自定义比较器
LinkedList<MatchOrder> 同价格档位订单队列 插入 O(1)、遍历 O(n) 保持 FIFO 时间优先、支持迭代器删除
Long2ObjectHashMap 订单 ID 到订单的映射 查找 O(1) Agrona 高性能集合，避免 Long 装箱
BigDecimal 价格、数量 - 精确计算，避免浮点数精度问题
HashMap<String, OrderBook> 交易对到订单簿映射 查找 O(1) 可考虑换成 Object2ObjectHashMap

Phase 1 (预撮合):
  • 使用 deepCopy() 复制订单
  • 模拟成交过程，不修改原订单簿
  • 收集需要 STP 撤销的订单到 fokPendingRemoves
  • 判断是否可完全成交
Phase 2 (真实撮合):
  • 先执行 STP 撤销
  • 再执行真实撮合
  • 保证 FOK 原子性语义
// Phase 1: 预撮合
MatchOrder copyOrder = newOrder.deepCopy();
boolean exit = this.fokPhase1(copyOrder, depthLine, orderBook,
    fokPendingRemoves, fokCancelReasonMap);

// 判断是否可完全成交
if (copyOrder.getOrderStatus() != OrderStatusEnum.FULL_FILLED) {
    newOrder.cancel(CancelReasonEnum.FOK_NOT_FULLFILL_CANCEL);
    return;
}

// Phase 2: 真实撮合
for (MatchOrder needCancelOrder : fokPendingRemoves) {
    needCancelOrder.cancel(fokCancelReasonMap.get(needCancelOrder.getOrderId()));
    orderBook.removeOrder(needCancelOrder);
}
fokTryMatchInstantly(newOrder, orderBook);
